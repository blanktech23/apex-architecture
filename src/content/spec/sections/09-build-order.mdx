Timeline: 26--28 Weeks (Optimistic) / 34--38 Weeks (Realistic)


The original 26--28 week estimate assumes zero rework, no integration delays, and maximum parallelism. Based on industry benchmarks for solo/duo developer teams building distributed systems, a 1.5--2x calendar time overrun is standard. The wave structure below reflects the 26--28 week optimistic schedule; add 30--40% buffer for realistic planning.


The original 14-week plan prioritized speed over production-readiness. This revised plan adds CI/CD, testing, security hardening, and monitoring from Day 1 --- and launches with 2--3 agents instead of 7 to ensure quality. Remaining agents are built post-launch. Every successful construction SaaS (Procore, Buildertrend) started by solving ONE problem well. Additional weeks allocated for Supabase RLS optimization (Wave 1), Stripe integration complexity (Wave 2), and risky third-party integrations in Wave 5 (JobTread Pave, QuickBooks OAuth).


Timeline Risk


The 26--28 week build order assumes two full-time developers with no context-switching. Historical data on projects of this complexity suggests 1.5--2x calendar time overrun is common. The wave structure is designed so each wave ships independently usable functionality --- if timelines slip, Wave 3 (first demo) remains the hard gate for sales activity.

During the rolling deploy window where VPS nodes run different code versions, route signatures must be backward-compatible --- enforce via CI check that diffs the exported `AppType` across commits.

### Wave 2a: AI Gateway + Agent Executor (Week 6--7)


Wave 2 Density Warning (SCALE-04)


Wave 2 scope is the densest in the plan. Consider splitting into Wave 2a (AI gateway + agent executor framework, 2 weeks) and Wave 2b (Stripe + error patterns + observability, 2--3 weeks). Stripe webhook integration (idempotency, replay, event ordering, subscription state machine) is routinely underestimated.


- Vercel AI Gateway configuration + AI SDK middleware (budgetEnforcement, costTracking, logging) + model routing via provider registry + **model escalation safety net** (see AI-26 below)

- Agent executor framework (thorchh/agent-starter base + Vercel AI SDK (AI SDK v6) for tool calling, streaming, structured output) + **`AgentPluginFactory` registration API** (Section 6) + **13-event streaming protocol** (AI-25, Section 7b)

- Agent template system (Slate "Construction Remodeling" = first template)

- NATS event bus (pub/sub per tenant, subject namespacing) + BullMQ job queue. *Initial concurrency bounds (validated via Wave 2 load testing): Per-node worker concurrency: 10 concurrent jobs per priority level. Per-tenant concurrent job limit: 5 simultaneous agent executions. Priority mapping: escalations = high, user-triggered = medium, scheduled/cron = low.* **Write coordination:** With up to 5 concurrent agent executions per tenant, race conditions on shared state are possible. BullMQ does not deduplicate jobs by default. Mitigations: (1) Use deterministic jobId values scoped to `${tenantId}:${agentId}:${triggerType}:${windowId}` where windowId is a time-bucket (floor to 5-minute intervals for scheduled jobs) to prevent duplicate queuing. (2) Document the consistency model: dashboard data is eventually consistent with a bounded staleness of the last agent execution cycle. Display `last_updated` timestamps on dashboard panels. (3) For user-triggered executions, use a Valkey SET NX lock to prevent double-submission of the same agent for the same tenant within a 30-second window.

> **Spec Amendment (CRITICAL) (Attack Audit 2026-03-22):** Every BullMQ queue MUST have `defaultJobOptions: { removeOnComplete: { count: 1000, age: 3600 }, removeOnFail: { count: 5000 } }`. Without this, Valkey memory grows linearly --- 4.5M jobs = ~10GB. This is the #1 BullMQ production incident. Apply to every `new Queue()` instantiation across agent queues, memory extraction queues, scheduled job queues, and data import queues.

> **Spec Amendment (Attack Audit 2026-03-22):** Enforce per-tenant limit via BullMQ `limiter: { max: 5, duration: 1000, groupKey: 'tenantId' }` on worker, OR Valkey-based semaphore checked before processing. The current "5 simultaneous" limit needs a concrete enforcement mechanism --- without one of these, the limit is aspirational, not enforced.

> **Spec Amendment (Attack Audit 2026-03-22):** Multi-tenant AI concurrency limits: Global concurrency limit: 50 concurrent agent executions per VPS node (100 total across dual VPS). Per-tier limits: Starter = 2 concurrent, Professional = 5 concurrent, Enterprise = 10 concurrent agent executions. When a tenant is at capacity, new agent triggers queue in BullMQ with estimated wait time pushed via Socket.io (`agent:queue_position` event with `{ position: number, estimatedWaitMs: number }`). Per-tenant storage quotas: Starter = 100MB (memories + embeddings + uploads), Professional = 1GB, Enterprise = 10GB. Quota enforcement: check before embedding generation and file upload. Quota exceeded -> `TENANT_QUOTA_EXCEEDED` error + in-app notification suggesting plan upgrade.

- **Embedding pipeline (AI-16b):** OpenAI `text-embedding-3-small` ($0.02/1M tokens). Chunking: 512-token chunks with 50-token overlap. Embeddings generated on document write/update via BullMQ background job. Index: HNSW for = 80% human approval rate on shadow responses.

- **RAGAS evaluation pipeline:** Automated evaluation of RAG quality using RAGAS framework, run during shadow mode. Minimum thresholds before go-live: faithfulness >= 0.90 (responses grounded in retrieved context), answer relevancy >= 0.85 (responses address the actual question), context precision >= 0.80 (retrieved chunks are relevant to the query). Evaluation runs against a curated test set of 100+ question-answer pairs covering all 5 KB categories.

### Data Import & Export (Wave 4)


Data Portability Requirement


Data import and export are both Wave 4 deliverables. Import reduces onboarding friction for customers migrating from spreadsheets or other tools. Export is required for CCPA data portability compliance and builds customer trust ("your data is yours").


#### Data Import


- **CSV import adapter:** Support CSV import for projects, contacts, and estimates. Mapping UI allows the user to map CSV columns to canonical domain model fields (Section 3c / Wave 1 domain interfaces).

- **Validation:** Pre-commit validation with error report --- show row-level errors (missing required fields, type mismatches, duplicate detection) before any data is written. User reviews and confirms before import executes.

- **Historical backfill:** For JobTread customers, initial data sync via Pave API pulls historical projects, contacts, and estimates into the local data mirror (Section 7a Anti-Corruption Layer). This is part of the onboarding flow, not a separate import step.

- **Implementation:** BullMQ background job for large imports (>100 rows). Progress indicator via Socket.io real-time updates. Rollback on critical failure (transactional import per batch).


#### Data Export


- **"Download all my data" button:** Available in account settings for Owner/Admin roles. Exports all tenant-scoped data as a ZIP archive containing JSON (machine-readable) + CSV (human-readable) files.

- **Scope:** All data from tenant-scoped tables --- projects, contacts, estimates, agent execution history, conversation threads, drafts, escalations, audit logs.

- **Implementation:** BullMQ background job assembles the archive, uploads to DO Spaces with a signed URL (24-hour expiration). User notified via email + in-app notification when export is ready. Rate limited to 1 export per 24 hours per tenant.

- **CCPA compliance:** This export capability satisfies CCPA data portability requirements. Combined with the data deletion capability in the tenant offboarding flow (Section 5), this completes the CCPA data rights implementation.
