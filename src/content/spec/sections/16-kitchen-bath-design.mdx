A browser-based 3D kitchen and bathroom design tool that replaces legacy Windows-only CAD software with an AI-powered, collaborative platform. This module leverages three AI engines in a closed feedback loop to generate, validate, and evaluate designs in real time --- bringing professional-grade kitchen design to the browser.


**Deployment model:** The K&B design editor is **embedded directly within the Design Agent agent's chat window** in the Kiptra dashboard --- not a separate app or subdomain. The designer converses with the Design Agent ("Create an L-shaped kitchen, 12x14, white shaker cabinets, quartz countertops"), and the 3D viewport renders inline alongside the chat. This preserves the existing auth flow (no separate login), shares Zustand state with the dashboard, and keeps the design workflow within the agent interaction paradigm that defines the entire Kiptra platform. The `kb-editor` package exports a `` React component consumed by the agent chat UI.


### 16.1 Overview & Strategic Rationale


The Kitchen & Bath AI Design Tool is a ground-up reimagining of how kitchen and bathroom spaces are designed. It replaces Cyncly's 2020 Design --- the incumbent tool that dominates the K&B industry --- with a modern, browser-based alternative powered by AI.


**Key insight:** 2020 Design is Windows-only, crash-prone, and costs $2,500--$3,000/year after private equity extraction. Designers hate it but remain locked in because manufacturer catalogs are only available through its ecosystem. Catalog data is available through manufacturer relationships, aggregators (KitchenDEV, Kitchen365), and public spec sheets. While Cyncly's ecosystem creates friction, underlying product data is not exclusively controlled by any single vendor --- meaning the lock-in is largely artificial.


**Why Kiptra builds this:**


- **Co-founder validation:** Adam runs a K&B design and remodel business with direct manufacturer relationships. He has confirmed demand from peers who would switch immediately given a viable alternative.

- **Market disruption:** Legacy CAD tools (Revit, 2020 Design, ProKitchen) are desktop-bound, expensive, and have not innovated in a decade. A browser-based, AI-powered tool with real-time collaboration is a generational leap.

- **Catalog data is accessible:** Manufacturer product data (dimensions, finishes, pricing, 3D models) is available through multiple channels --- dealer portal exports, FTP data drops, partnership-negotiated access (requires Adam's manufacturer relationships), 2020 Design format files, PDF spec sheets, and aggregators like KitchenDEV/Kitchen365.

- **AI makes it possible now:** LLMs can generate spatial layouts from natural language. SMT solvers can formally verify building code compliance. Vision models can evaluate rendered designs for aesthetics and ergonomics. None of this was feasible two years ago.

> **Spec Amendment (Attack Audit 2026-03-22):** AI loop error handling matrix is undefined. Required failure modes:
> - **Claude invalid JSON:** Retry once with stricter prompt. If second attempt fails, return schema error to designer.
> - **Claude timeout (>30s):** Retry once. If second timeout, fall back to nearest template layout for room dimensions.
> - **Z3 hangs (>5s):** Terminate WASM Worker via `worker.terminate()`. Apply `rlimit`-style timeout on server. Return partial validation with warning.
> - **Gemini timeout (>15s):** Skip visual evaluation for this round. Proceed with Z3-validated layout (score = N/A).

#### Layout Generation (Claude Sonnet/Opus)


Natural language input is converted to structured JSON kitchen layouts using a cabinet "runs" abstraction --- not individual (x,y) coordinates. A run is a sequence of cabinets along a wall segment. A deterministic function converts runs to precise coordinates, eliminating the geometric errors that plague direct LLM coordinate generation.

> **KB Amendment:** Appliance dimensions must be specified or selected from catalog BEFORE layout generation. This prevents the #1 design regret (appliance-cabinet fit issues). The AI loop should reject layout requests without appliance specifications.



AI Spatial Reasoning Limitation


LLM spatial reasoning has known limitations --- models struggle with precise geometric placement and distance estimation. The cabinet-run abstraction with deterministic coordinate resolution works around these limitations by keeping LLMs in the semantic domain (which cabinets, in what order) rather than the geometric domain (exact x,y coordinates). The Z3 solver catches geometric constraint violations that LLMs cannot reliably avoid, ensuring NKBA compliance regardless of LLM output quality.

// Claude outputs cabinet runs, not coordinates
interface CabinetRun {
 wallId: string;
 startOffset: number; // inches from wall start
 direction: &apos;left-to-right&apos; | &apos;right-to-left&apos;;
 cabinets: RunCabinet[]; // sequential, no gaps
}

// Deterministic function --- no LLM geometry errors
function resolveRunToCoordinates(
 run: CabinetRun,
 walls: Wall[]
): PlacedCabinet[] {
 const wall = walls.find(w => w.id === run.wallId);
 let cursor = run.startOffset;
 return run.cabinets.map(cab => {
 const pos = wallPointAtOffset(wall, cursor);
 cursor += cab.width;
 return { ...cab, x: pos.x, y: pos.y, rotation: wall.angle };
 });
}

#### Constraint Validation (Z3 SMT Solver)


The Z3 SMT solver provides formal NKBA compliance validation. Of the 31 NKBA Kitchen Planning Guidelines, 24 are fully encodable as mathematical constraints. When a layout fails validation, Z3 returns UNSAT cores --- the minimal set of constraints that conflict --- telling Claude exactly what to fix.

#### Micro-Benchmark Details


The GPU benchmark renders a rotating scene with 10,000 textured triangles (a single `BufferGeometry`) for exactly 120 frames, measuring actual frame delivery time via `performance.now()`. Total benchmark time is ~2 seconds at 60fps. The benchmark canvas is 256x256 pixels (offscreen) to minimize overhead. Result is median FPS across the 120 frames, excluding the first 10 frames (warm-up). The benchmark is **not** Three.js --- it uses raw WebGL to avoid loading the Three.js bundle for devices that will ultimately be assigned Tier 3 or 4.

GPU Memory Monitoring Limitation


There is no reliable cross-browser API for querying actual GPU memory usage. `WEBGL_debug_renderer_info` exposes the GPU model name (from which memory can be estimated) but not live memory consumption. The `performance.memory` API (Chrome-only) reports JS heap, not GPU memory. Our approach: maintain a running estimate of allocated texture bytes based on what the application loads (tracked in a `TextureMemoryTracker` singleton). When estimated usage exceeds 70% of the tier's texture memory budget, begin evicting least-recently-used textures and downgrading LOD levels. This is a best-effort heuristic, not a guarantee.


**Mobile Safari GPU constraint:** WebGL contexts on mobile Safari are limited to ~256MB GPU memory. Mitigation: `TextureMemoryTracker` enforces a 200MB hard cap on iOS, aggressive LOD downgrading at 150MB, and KTX2 compression reduces texture memory by 4--6x compared to uncompressed RGBA.


**Decisions locked:**


**1. Four tiers, not three.** Tier 3 (2D only) and Tier 4 (view only) are distinct because jobsite tablet users need to *view* designs even if they cannot edit them. A "sorry, your device isn't supported" message is unacceptable for a construction platform.


**2. Detection before bundle load.** The ~3 KB detection script runs before React hydration to avoid downloading 450 KB of Three.js on devices that will never use it. This is a hard requirement --- not an optimization.


**3. Serverless GPU rendering, not CPU software rendering.** The server-side render uses Modal.com serverless GPU (NVIDIA A10/L4) with `three-gpu-pathtracer` running headless. Rationale: serverless GPU costs ~$0.001 per render ($0.0003/sec, 1--3 sec per 4K render) with zero idle cost (scale to zero, per-second billing). This is 5--10x faster than CPU software rendering (1--3 sec vs 5--15 sec), produces real GPU quality (hardware-accelerated ray tracing), and has no impact on VPS CPU/memory budget. Monthly cost at 10K renders: ~$10--30. Tiered escalation to Blender Cycles (photorealistic) and V-Ray App SDK (premium) as demand warrants.

#### Markup, Margin, and Labor Injection


The mapping layer applies three cost transforms before producing estimate line items. All rates are tenant-configurable via `tenants.estimate_config` JSONB.

 contingencyPercent: number; // e.g., 0.10 (10% contingency)
 laborHoursOverrides: Record; // override default hrs

// Transform: BomItem -> EstimateLine
function mapBomItemToEstimateLine(
 item: BomItem,
 config: EstimateConfig,
 categoryMap: Record
): EstimateLine {
 const markup = config.categoryMarkupOverrides[item.category]
 ?? config.defaultMarkupPercent;
 const laborHours = config.laborHoursOverrides[item.category]
 ?? DEFAULT_LABOR_HOURS[item.category];
 const materialCost = item.quantity * item.unitCost;
 const materialWithMarkup = materialCost * (1 + markup);
 const laborCost = laborHours * item.quantity * config.laborRatePerHour;

 return {
 category: categoryMap[item.category],
 description: item.name,
 sku: item.sku,
 quantity: item.quantity,
 unitOfMeasure: item.unitOfMeasure,
 materialCost: materialWithMarkup,
 laborCost,
 totalCost: materialWithMarkup + laborCost,
 sourceRef: { bomItemId: item.id, productId: item.productId },
 };
}

Core Value Proposition


This bridge is the reason the K&B tool exists inside Kiptra rather than as a standalone product. Without it, a designer uses 2020 Design to create a layout, manually re-enters every cabinet SKU into a spreadsheet or JobTread to build an estimate, then manually creates a project in their PM tool. Three separate tools, three data entry passes, three opportunities for transcription errors. With the bridge, the designer finalizes a 3D layout and clicks one button. The BOM is extracted from the scene graph, prices are resolved from the catalog, markup and labor are applied per the contractor's configured rates, and a complete estimate draft lands in their approval queue --- traceable back to the exact design version that produced it. The estimate syncs to JobTread or Procore via the same ACL that handles every other integration. This **design -> estimate -> project** pipeline in a single platform is the moat: it eliminates the data re-entry that costs K&B contractors 2--4 hours per project and removes the pricing errors that erode margins on 30%+ of remodeling jobs.


> **KB Amendment (2026-03-21):** 40-50% of a kitchen/bath remodel estimate consists of items NOT in the 3D design's BOM: demolition ($1,500-5,000), plumbing rough-in ($2,500-4,000), electrical ($1,800-3,500), permits ($200-2,000), floor protection, dumpster, painting, and contingency (10-20%). The estimate engine must auto-suggest non-BOM item templates based on project type (kitchen vs bath) with tenant-configurable local cost defaults.


Implementation Sequencing


The BOM -> Estimate bridge depends on three prerequisites: (1) the canonical domain model's `Estimate` and `EstimateLine` interfaces (Section 3c, Wave 1), (2) the `create_estimate_draft` tool in the Sales Agent (Wave 2), and (3) the `kb_catalog_products` table populated with at least one manufacturer's catalog data (Month 3 of K&B build). The bridge itself is a Month 4--5 deliverable --- after BOM extraction exists (Month 3) and after the Sales Agent is operational (Wave 2). The NATS event flow and ACL sync are already built by the time this bridge ships.

### 16.5 NKBA Design Standards Enforcement


The National Kitchen & Bath Association publishes 31 Kitchen Planning Guidelines covering clearances, landing areas, storage minimums, ventilation, and safety.


#### Two-Tier Validation


**Tier 1 --- TypeScript Rule Engine (real-time, client-side):** Simple distance and clearance checks per frame. Violations highlighted in red with inline measurements showing the shortfall.


**Tier 2 --- Z3 SMT Solver (WASM or server):** Full formal optimization and feasibility checking. Used to validate Claude's generated layouts and return UNSAT cores identifying exactly which constraints conflict. Runs in ~100--500ms.

-- 4. Catalog products (SKUs) with full-text + vector search

> **KB Amendment:** Changed from VECTOR(384) to halfvec(768). BGE-base at 768 dimensions with halfvec quantization provides adequate discrimination at half the storage cost of full float vectors.

CREATE INDEX idx_catalog_search ON kb_catalog_products USING GIN(search_vector);
CREATE INDEX idx_catalog_embedding ON kb_catalog_products USING hnsw(embedding vector_cosine_ops);

);

#### 16.13.2 NKBA Intellectual Property
**Copyright:** NKBA Kitchen & Bath Planning Guidelines are copyrighted and sold commercially. Dimensional facts (e.g., "36 inches minimum clearance") are NOT copyrightable (Feist Publications v. Rural Telephone, 1991). Creative expression (guideline text, diagrams, commentary) IS copyrightable.

**Trademark:** "NKBA" is a registered trademark. Cannot say "NKBA Certified" or "NKBA Approved" — implies endorsement.

**Required mitigations:**
- Encode rules as functional code using dimensional values only — never copy NKBA guideline text verbatim
- UI must say "designed following industry dimensional guidelines" NOT "NKBA Compliant"
- Add non-affiliation disclaimer: "Not affiliated with or endorsed by the National Kitchen & Bath Association"
- The mockup's "NKBA Compliant" badge must be reworded to "Code-Compliant" or "Clearance Verified"

#### 16.13.3 Manufacturer Catalog Data
Dealer net pricing (multipliers, volume tiers) is often treated as trade secret by manufacturers. Unauthorized storage or display may violate dealer agreements.

**Required mitigations:**
- Written permission from each manufacturer before storing pricing data
- Dealer net costs visible ONLY to the authenticated contractor who owns the dealer account — never exposed publicly
- Data licensing agreement template for manufacturer onboarding
- Price data must be deletable per manufacturer request (contractual right)

#### 16.13.4 OFDA Standard Correction
OFDA (Office Furniture Dealers Alliance) XML is for commercial office furniture ordering, NOT kitchen cabinets. Kitchen cabinet manufacturers use proprietary ordering formats (manufacturer-specific portals, CSV/Excel uploads). References to "OFDA XML" in this spec should be understood as "manufacturer-specific order format" — the actual format varies by manufacturer (KraftMaid, Merillat, Yorktowne each have their own).

#### 16.13.5 Data Privacy (CCPA / State Privacy Laws)
Floor plans with customer addresses, project details, and pricing constitute personal information under CCPA and 15+ state privacy laws.

**Required mitigations:**
- Privacy policy covering design data collection, storage, and sharing
- Data Subject Access Request (DSAR) handling process
- Multi-tenant data isolation (already implemented via Supabase RLS)
- Data retention policy with customer deletion rights
- Design data encrypted at rest

Generating construction estimates in software does not require a contractor's license in most states, provided estimates are framed as "approximate cost ranges" not "bids" or "quotes."
