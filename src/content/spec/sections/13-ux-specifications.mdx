This section defines the product-level UX specifications that complement the architecture described in Sections 1--12. Each subsection covers a discrete functional area: screen layouts, user stories, data flows, RBAC rules, error states, and edge cases. These specs serve as the implementation contract between design and engineering.


#### Global UX Conventions


The following conventions apply across all screens unless a subsection explicitly overrides them.


- **Navigation shell:** All authenticated pages render inside a shared layout with a **collapsible left sidebar** (240px expanded, 64px icon-only collapsed) and a **top bar**. The sidebar contains: logo, primary nav links (Dashboard, Agents, Customers, Projects, Escalations (badge), Approvals (badge), Reports, Business OS, Settings), seat/plan badge, and a collapse toggle. **Contextual sidebar swap:** On `/bos/*` routes, the sidebar replaces main navigation with Business OS grouped navigation (Home, Execution, Strategy, People) and a "Back" button to return to the main nav. On `/settings/*` routes, the sidebar shows settings sub-navigation if applicable. The top bar contains: breadcrumbs (left), notification bell with unread badge (center-right), and user avatar dropdown (right: profile, settings, sign out). On mobile ( link to agent setup; "No integrations connected" -> "Connect your first integration" button; "No team members yet" -> "Invite your first team member").

- **Keyboard accessibility:** All interactive elements are reachable via Tab. Focus indicators use the accent color outline (`2px solid var(--accent)`). Enter/Space activate buttons and toggles. Escape closes modals and dropdowns. Arrow keys navigate within dropdowns, date pickers, and table rows. Skip-to-main-content link is the first focusable element on every page.

- **Help access:** A persistent Help Bot button (`MessageCircleQuestion` icon) in the bottom-right corner of every authenticated page. Clicking opens a slide-out panel with two modes: Quick Help (contextual suggestions matched to current route, search bar, quick links) and AI Chat (Haiku-powered, streamed responses with deep links). Owner/Admin also see a "Schedule a call" option. Full specification in Section 13m.

> **Spec Amendment (Attack Audit 2026-03-22):** Use `h-dvh` not `h-screen` for all full-height layouts. Mobile Safari 100vh includes address bar, causing overflow. `h-dvh` adjusts dynamically.

> **Spec Amendment (Attack Audit 2026-03-22):** Internationalization deferred. US English only for MVP. Date formatting: explicit `en-US` locale. Currency: `Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' })`. When needed: adopt `next-intl` with ICU message format.

### 13-UI. Visual Design System


This subsection defines the foundational visual language for all Kiptra AI screens. Every screen spec in Section 13 inherits these tokens, components, and conventions unless explicitly overridden. The system is built on **shadcn/ui + Tailwind CSS + Radix UI** primitives.


#### Design Foundation


**Framework:** shadcn/ui + Tailwind CSS + Radix UI primitives


**Philosophy:** Clean, professional, high-contrast for readability on job sites (bright sunlight, dusty screens). Prioritize clarity over decoration. Construction professionals need to glance and understand --- no ambiguity.

> **Spec Amendment (Attack Audit 2026-03-22):** Design token pipeline (deferred post-MVP): Figma Variables → Tokens Studio → Git PR → Style Dictionary → CSS custom properties → `tailwind.config.ts`. For MVP, tokens manually maintained. Token names MUST follow semantic naming (`--color-primary` not `--blue-800`) for future pipeline adoption.


#### Color Palette


All colors are defined as CSS custom properties and mapped into `tailwind.config.ts`. Dark mode is deferred to post-launch. Token architecture is ready (all colors reference CSS custom properties) but dark mode color values have not been designed. Estimated effort: 1--2 days design + 0.5 day implementation to swap custom property values via `prefers-color-scheme` or a manual toggle.

- **Card gap:** `gap-4` (16px) in grid layouts

#### Data Visualization (Recharts)


- **Chart colors:** Primary Blue, Accent Amber, Success Green --- max 3 colors per chart for clarity

- **Chart types:** Line charts for daily usage trends and cost over time. Bar charts for per-agent execution breakdown and per-user activity. **No donut/pie charts** (hard to read; use bar charts instead).

- **Grid lines:** Light gray (`#E2E8F0`), horizontal only

- **Labels:** Always-visible axis labels; tooltip on hover with exact values

- **Responsive:** Charts resize to container width. On mobile (1024px | Full sidebar, multi-column layouts, data-dense tables | |


**Construction-specific:** All touch targets are minimum 44x44px (WCAG). Action buttons on mobile are oversized for gloved or dirty hands.


#### Responsive Table Pattern


All data tables use a `ResponsiveDataTable` wrapper component that renders as a standard `` on desktop (>768px) and switches to a vertically stacked card layout on mobile (` and `` tags)


Each error boundary displays: a user-friendly error message, a "Try again" button (calls `reset()`), and a "Go to dashboard" fallback link. Error details are reported to Sentry automatically via the `useEffect` hook in the error component.


#### Accessibility (WCAG 2.1 AA)


- **Color contrast:** All text meets 4.5:1 ratio. Verified: `#0F172A` on `#FFFFFF` = 15.4:1; `#334155` on `#FFFFFF` = 8.6:1.

- **Color independence:** Status indicators use icon + color + label (never color alone). Usage meters show percentage text alongside colored bar.

- **Focus indicators:** 2px blue outline on keyboard focus (not visible on mouse click via `:focus-visible`)

- **Screen reader:** All images have `alt` text, all icons have `aria-label`, all form inputs have associated `` elements.

- **Reduced motion:** Respect `prefers-reduced-motion` --- disable all transitions and animations.

- **Live regions for real-time updates:** All Socket.io-driven content updates (agent status changes, escalation alerts, execution progress) must use `aria-live="polite"` regions so screen readers announce changes without requiring page refresh.

- **Role attributes for custom widgets:** Custom components not using native HTML elements must include appropriate `role` attributes: React Flow canvas (`role="img"` with `aria-label`), agent status cards (`role="status"`), and toast notifications (`role="alert"`).

- **Accessible chart names:** All Recharts visualizations must include `aria-label` describing the chart content (e.g., "Monthly revenue trend, January through March 2026") and a visually hidden data table fallback for screen reader users.

- **Keyboard patterns for TanStack Table:** Arrow key navigation between cells, Enter to activate row actions, Escape to exit cell focus. Column sort toggleable via Enter on header cells.

- **320px reflow:** All content must be usable at 320px viewport width (WCAG 1.4.10) without horizontal scrolling for text content. Data tables may scroll horizontally but must provide a card-based alternative view on mobile (see Responsive Table Pattern below).

### 13a. Login Page


Routes: `/login`, `/login/mfa`


#### User Stories


- Returning user signs in with email/password.

- Returning user signs in via Google or Microsoft OAuth.

- MFA-enabled user enters TOTP code after password authentication.

- User clicks "Forgot password?" to start the reset flow.


#### Screen: `/login`


**Layout:** Centered card (max-width 420px). Company logo + tagline at top. Clean, single-column form.


- **Email input** --- standard text field with validation.

- **Password input** --- with show/hide toggle (eye icon).

- **"Sign in" primary button** --- full-width, accent color.

- **Divider** --- "or continue with" horizontal rule with centered text.

- **OAuth buttons** --- Google and Microsoft, each with provider icon. Outlined style to distinguish from primary CTA.

- **"Forgot password?" link** --- below password field, routes to `/forgot-password`.

- **"Don't have an account? Contact sales"** --- Phase 1: no self-serve signup. Phase 2: becomes a `/signup` link.


#### Error States


| Condition | Message Shown | Notes | |


| Invalid email or password | "Invalid email or password." | Generic message --- never reveal which field is wrong (enumeration prevention). | |

| Account locked (5 failed attempts) | "Account locked. Try again in 15 minutes or contact support." | Lockout duration: 15 min. Counter resets on successful login. Audit log entry written on lock. | |

| OAuth provider error | "Could not connect to [Provider]. Please try again." | Log full error server-side; surface only generic message to user. | |

| Network error | "Network error. Check your connection and try again." | Client-side detection via fetch failure / timeout. | |


#### Screen: `/login/mfa`


Shown after successful email/password authentication when MFA is enabled on the account.


- **"Enter your 6-digit code" heading** --- clear instruction.

- **6-digit TOTP input** --- individual digit boxes with auto-advance on each keystroke. Auto-submit on 6th digit.

- **"Verify" button** --- primary CTA, also triggered by auto-submit.

- **"Use a recovery code instead" link** --- switches to a single text input for the recovery code.

- **"Back to login" link** --- clears partial auth state, returns to `/login`.


#### MFA Error States


| Condition | Message Shown | Notes | |


| Invalid code | "Invalid code. Please try again." | Clear input, refocus first digit. | |

| Expired code (TOTP window passed) | "Code expired. Enter your current code." | TOTP window: 30s. Accept current + previous window to handle clock skew. | |

| Too many attempts (5 failures) | "Too many attempts. Please sign in again." | Invalidate partial auth session. Redirect to `/login`. Audit log entry. | |


#### Data Flow


**Reads:** `users` table (better-auth), `tenant_members` for org membership lookup.


**Writes:** Session token (better-auth managed), audit log entry (login success/failure, IP, user agent).


**Post-auth routing:** If user belongs to a single organization -> redirect to `/dashboard`. If multiple organizations (future) -> org picker screen before dashboard.


**Multi-org session scoping:** Selected `organization_id` is embedded in the session token at selection time. Switching tenants issues a NEW session token (not a client-side header swap). `withTenantContext()` derives tenant_id exclusively from the session token. **Multi-tab session model:** better-auth sessions are cookie-based and SHARED across all tabs in the same browser. Logging out in one tab logs out all tabs. The plan does NOT claim per-tab session isolation. For multi-org access in separate tabs, users must use different browser profiles (or the future multi-org session switcher must re-scope the shared session cookie).


#### Edge Cases


| Scenario | Behavior | |


| **Multiple organizations** | Org selector shown post-auth. User picks which org to enter. Session scoped to selected org. | |

| **Account locked** | "Try again in 15 minutes" message with support contact link. No hint about which credential failed. | |

| **OAuth-only account** | Password field hidden. Only OAuth buttons rendered. If user somehow reaches password form, show "This account uses Google/Microsoft sign-in." | |

| **Social login email matches existing account** | better-auth handles account linking with a confirmation dialog: "An account with this email already exists. Link your [Provider] account?" | |

### 13b. Password Reset Flow


Routes: `/forgot-password`, `/forgot-password/sent`, `/reset-password?token=xxx`, `/reset-password/success`


#### User Stories


- User who forgot their password initiates a reset via email and sets a new password.


#### Screen: `/forgot-password`


- **Email input** --- single field.

- **"Send reset link" button** --- primary CTA.

- **"Back to login" link** --- returns to `/login`.


**Security:** Always shows the success screen regardless of whether the email exists in the system. This prevents email enumeration attacks. The server returns 200 OK in all cases.


#### Screen: `/forgot-password/sent`


- **"Check your email" message** --- includes the email address submitted (masked: j***@example.com).

- **"Resend link" button** --- rate limited to 1 request per 60 seconds. Button disabled with countdown timer during cooldown.

- **"Back to login" link**.


#### Screen: `/reset-password?token=xxx`


- **Token validation on page load** --- server verifies token existence, expiry (1 hour), and single-use status before rendering the form.

- **New password input** --- with strength meter (zxcvbn or equivalent). Shows requirements inline.

- **Confirm password input** --- must match.

- **"Reset password" button** --- primary CTA. Disabled until both fields valid and matching.


**Token expiry:** 1 hour from generation. If the token is invalid or expired, show an error page with a "Request new link" CTA that routes back to `/forgot-password`. Never reveal why the token failed (expired vs. invalid vs. already used) --- just "This link is no longer valid."


#### Screen: `/reset-password/success`


- **"Password updated" confirmation** --- clear success state.

- **"Sign in" button** --- routes to `/login`.

- **All existing sessions revoked** --- server-side, on password change via explicit `auth.api.revokeAllSessions()` call (better-auth does NOT auto-revoke sessions on password change). User must re-authenticate on all devices.


#### Edge Cases


| Scenario | Behavior | |


| **Token already used** | "This link is no longer valid. Request a new one." --- same message as expired (no information leakage). | |

| **Token expired** | "This link is no longer valid." + "Request new link" CTA. | |

| **Email not found** | Still shows "Check your email" on `/forgot-password/sent`. No email sent server-side. | |

| **OAuth-only user** | Email sent says: "Your account uses Google/Microsoft sign-in. Use that provider to access your account." No reset link included. | |

| **Same password as current** | Optionally block with "New password must be different from your current password." Configurable per tenant policy. | |

### 13c. Invitation Acceptance Flow


Routes: `/invite/accept?token=xxx`, `/invite/register?token=xxx`, `/invite/success`


#### User Stories


- Invited team member accepts an invitation and creates a new account.

- Existing user accepts an invitation to join an additional organization with one click.


#### Screen: `/invite/accept?token=xxx`


Landing page for all invitation links. Shows invitation details and branches based on user state.


- **Invitation details** --- Company name, inviter name, role being assigned (e.g., "Sarah invited you to join Slate Design as a Project Manager").

- **If logged in with matching email** -> "Accept Invitation" button (one-click join). No further steps required.

- **If not logged in but account exists** -> "Sign in to accept" button. Routes to `/login` with `redirect=/invite/accept?token=xxx`. After login, returns here for one-click accept.

- **If no account exists** -> "Create account to accept" button. Routes to `/invite/register?token=xxx`.


#### Screen: `/invite/register?token=xxx`


Registration form scoped to the invitation. The token is validated on load --- if invalid/expired, show error instead of form.


- **Email** --- pre-filled from invitation record, **not editable**. Displayed as read-only text.

- **Full name input** --- required.

- **Password input** --- with strength meter (same component as reset flow). Shows requirements inline.

- **OAuth buttons** --- "or sign up with Google / Microsoft" as alternative to password creation.

- **"Create account & join [Company]" button** --- primary CTA. Company name pulled from invitation record.


#### Screen: `/invite/success`


- **"You've joined [Company] as [Role]" message** --- confirmation with company name and assigned role.

- **Brief role description** --- one sentence explaining what the role can do (e.g., "As a Project Manager, you can manage projects, view estimates, and coordinate with agents.").

- **"Go to Dashboard" button** --- primary CTA, routes to `/dashboard`.


#### Edge Cases


| Scenario | Behavior | |


| **Invitation expired (72h)** | "This invitation has expired. Ask [Inviter Name] to send a new one." Inviter name pulled from invitation record. | |

| **Already accepted** | "You've already joined [Company]." + "Go to Dashboard" link. No error styling --- treat as success. | |

| **Invitation revoked** | "This invitation is no longer valid." No further detail (don't reveal revocation). | |

| **Email mismatch (logged in as different user)** | "This invitation was sent to [invited@email]. Sign in with that email or sign out first." Show both options. | |

| **Seat limit reached** | "[Company] has reached its member limit. Contact the account owner to upgrade." Invitation remains valid --- can be accepted once a seat opens. | |


#### Data Flow


**Reads:** Invitation record (`token`, `org_id`, `role`, `inviter_id`, `email`, `expires_at`, `status`). User session (if logged in).


**Writes:** `users` table (if new account created), `tenant_members` (new membership row with assigned role), invitation `status` updated to `accepted`, audit log entry (invitation accepted, by whom, when).


**External:** `better-auth` --- `organization.acceptInvitation()` handles the membership creation and role assignment atomically.

### 13d. Signup & Onboarding Flow


Phase Context


Phase 1 is founder-led onboarding (remote via Google Meet/Zoom, per Wave 4). Self-serve signup is deferred to Month 5--6 of the GTM timeline. This specification covers both Phase 1 (founder-assisted) and Phase 2 (self-serve) flows. Screens marked **(Phase 2 only)** are not built until the self-serve milestone.


#### User Stories


- **CEO/owner (Phase 1):** Completes account setup after founder-led sales call. Plan is pre-selected; payment is handled offline via ACH/bank transfer or via a pre-configured Stripe link (ACH recommended, cards accepted with surcharge).

- **CEO/owner (Phase 2):** Self-registers, selects a plan, pays via Stripe Checkout (ACH recommended, cards accepted with 2.9% + $0.30 surcharge), and begins onboarding without human assistance.

- **First user:** Sets up profile and security credentials during the onboarding wizard.

```
#### Usage Tracking Architecture


- **Ingestion:** Agent executions logged to `agent_executions` table with token counts (populated via AI SDK response metadata + costTrackingMiddleware)

- **Aggregation:** Daily aggregation job (BullMQ scheduled, `0 2 * * *`) computes per-tenant usage summaries

- **API:** Usage data exposed via internal endpoint: `GET /api/v1/usage?period=current` --- tenant_id derived from authenticated session via `withTenantContext()`. Never accept client-supplied tenant IDs in URLs for data queries.

- **Alerting:** Approaching-limit check runs hourly; triggers Socket.io notification + email at 80% and 100% thresholds


SQL
-- Daily aggregation query (runs via BullMQ cron)
INSERT INTO tenant_usage_daily (tenant_id, date, execution_count, tokens_input, tokens_output, cost_usd)
SELECT
 tenant_id,
 DATE(created_at) AS date,
 COUNT(*) AS execution_count,
 SUM(tokens_used) AS tokens_input,
 0 AS tokens_output,
 SUM(cost_usd) AS cost_usd
FROM agent_executions
WHERE created_at >= NOW() - INTERVAL '1 day'
GROUP BY tenant_id, DATE(created_at)
ON CONFLICT (tenant_id, date) DO UPDATE SET
 execution_count = EXCLUDED.execution_count,
 tokens_input = EXCLUDED.tokens_input,
 tokens_output = EXCLUDED.tokens_output,
 cost_usd = EXCLUDED.cost_usd;
```

##### Tour Trigger Hook

```typescript
'use client';
import { useEffect, useRef } from 'react';
import { usePathname } from 'next/navigation';

/**
 * Hook that evaluates whether a tour should start for the current user.
 * Loaded via dynamic import (ssr: false) to avoid hydration errors.
 *
 * Decision logic:
 * 1. Query tour_progress table for this user + tourId
 * 2. If status = 'completed' and tour_version matches current version -> skip
 * 3. If status = 'dismissed' and dismissed_at < 24h ago -> skip (cooldown)
 * 4. If another tour is already active (global ref) -> skip (max 1 per session)
 * 5. If trigger = 'first_login' and this is not first login -> skip
 * 6. If trigger = 'first_visit' and pathname !== triggerRoute -> skip
 * 7. If user role not in tour.roles -> skip
 * 8. Otherwise -> start tour
 */
function useTourTrigger(tourId: string, triggerRoute?: string): void {
  const pathname = usePathname();
  const hasTriggeredRef = useRef(false);

  useEffect(() => {
    if (hasTriggeredRef.current) return;

    async function evaluate() {
      // Fetch tour progress from API
      const res = await fetch('/api/v1/tours/progress');
      const progress = await res.json();
      const tourProgress = progress.find((p: { tour_id: string }) => p.tour_id === tourId);

      // Check completion, cooldown, active tour, route match
      // Implementation details per decision logic above
      // If all checks pass:
      hasTriggeredRef.current = true;
      const { driver } = await import('driver.js');
      await import('driver.js/dist/driver.css');
      // Initialize and start tour with steps from tour registry
    }

    evaluate();
  }, [tourId, triggerRoute, pathname]);
}
```

##### Tour Fatigue Prevention

```typescript
const TOUR_CONFIG = {
  /** Maximum automatic tours per login session */
  maxToursPerSession: 1,
  /** Hours between automatic tour triggers (completed or dismissed) */
  cooldownHours: 24,
  /** Tours that ignore cooldown on first login only */
  mandatoryTours: ['platform'] as const,
  /** Tours that trigger on first visit to their route (not on login) */
  selfTriggeredTours: ['bos', 'kb_canvas', 'integrations'] as const,
  /** Dismissal escalation thresholds */
  dismissalPolicy: {
    firstDismiss: { reshowAfterDays: 7 },
    secondDismiss: { reshowAfterDays: 30 },
    thirdDismiss: { neverAutoShow: true, contextualTooltipsOnly: true },
  },
} as const;
```

**Rationale:** Industry data shows maximum 1 automatic tour per session prevents tour fatigue (Chameleon 2025: self-triggered tours have 2--3x higher completion than auto-triggered). Self-triggered module tours (`first_visit`) outperform auto-triggered tours because the user has context and intent when they navigate to a module. The 24-hour cooldown prevents "Welcome to Module X!" from firing immediately after "Welcome to the Platform!" --- a pattern that causes permanent tour dismissal behavior.

##### SSR Safety

All tour components must be loaded client-side only:

```typescript
import dynamic from 'next/dynamic';

const TourProvider = dynamic(() => import('@/components/tours/TourProvider'), {
  ssr: false,
  loading: () => null, // Tours are non-essential --- no placeholder
});
```

This prevents `ReferenceError: window is not defined` during build and avoids hydration mismatches. Every tour library (Driver.js, Onborda, Shepherd, etc.) requires this pattern in Next.js App Router.

##### Element Targeting Strategy

Use `data-tour` attributes instead of CSS class selectors or IDs:

```tsx
<div data-tour="dashboard-overview" className={styles.dashboardCard}>
  {/* Dashboard content */}
</div>
```

**Rationale:** `data-tour` attributes survive CSS refactors, className changes, and component restructuring. CSS selectors and IDs break when styling is updated. All tour-targeted elements must include `data-tour` attributes --- enforce via ESLint custom rule or PR review checklist.

**Missing element handling:** Before starting a tour, validate that all step selectors resolve to DOM elements. If a selector is missing (lazy-loaded component, conditional render, API-dependent content), gracefully skip that step rather than crashing the tour. Use a `MutationObserver` with 5-second timeout for elements expected to load asynchronously:

```typescript
const waitForElement = (selector: string, timeout = 5000): Promise<Element | null> => {
  return new Promise((resolve) => {
    const el = document.querySelector(selector);
    if (el) return resolve(el);
    const observer = new MutationObserver(() => {
      const el = document.querySelector(selector);
      if (el) { observer.disconnect(); resolve(el); }
    });
    observer.observe(document.body, { childList: true, subtree: true });
    setTimeout(() => { observer.disconnect(); resolve(null); }, timeout);
  });
};
```

- **Widget location:** Right sidebar, collapsible. Persists across page navigations.
- **Progress bar:** Linear progress indicator showing "X of 7 complete" with percentage.

- **Dismissible:** After 80% completion (6 of 7 items) or explicit "Dismiss checklist" action. Dismissal is persisted to `tour_progress` table with `tour_id = 'checklist'`.

```sql
CREATE TABLE tour_progress (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  tour_id TEXT NOT NULL, -- 'platform', 'bos', 'kb_canvas', 'integrations', 'checklist'
  tour_version INTEGER NOT NULL DEFAULT 1,
  status TEXT NOT NULL DEFAULT 'not_started'
    CHECK (status IN ('not_started', 'in_progress', 'completed', 'dismissed')),
  current_step INTEGER NOT NULL DEFAULT 0,
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  dismissed_at TIMESTAMPTZ,
  dismiss_count INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, user_id, tour_id)
);

-- RLS: Users can only read/write their own tour progress within their tenant
ALTER TABLE tour_progress ENABLE ROW LEVEL SECURITY;

CREATE POLICY tour_progress_select ON tour_progress
  FOR SELECT USING (
    user_id = auth.uid()
    AND tenant_id = (SELECT tenant_id FROM tenant_members WHERE user_id = auth.uid() LIMIT 1)
  );

CREATE POLICY tour_progress_insert ON tour_progress
  FOR INSERT WITH CHECK (
    user_id = auth.uid()
    AND tenant_id = (SELECT tenant_id FROM tenant_members WHERE user_id = auth.uid() LIMIT 1)
  );

CREATE POLICY tour_progress_update ON tour_progress
  FOR UPDATE USING (
    user_id = auth.uid()
    AND tenant_id = (SELECT tenant_id FROM tenant_members WHERE user_id = auth.uid() LIMIT 1)
  );

-- Performance indexes
CREATE INDEX idx_tour_progress_user ON tour_progress(user_id);
CREATE INDEX idx_tour_progress_tenant_tour ON tour_progress(tenant_id, tour_id);
CREATE INDEX idx_tour_progress_status ON tour_progress(user_id, status)
  WHERE status IN ('in_progress', 'not_started');
```

#### 13l.6 Everboarding (Ongoing Onboarding)

Onboarding is not a one-time event. The following mechanisms keep users discovering value over their entire lifecycle:

**Tour Versioning:** The `tour_version` column on `tour_progress` enables re-triggering tours when content updates. When a tour definition's `version` field is incremented (e.g., after adding a new step for a new feature), users who completed the previous version see the tour again. Users who dismissed the previous version are NOT re-triggered --- they opted out intentionally.

**"What's New" Component:** A monthly changelog modal accessible via a "What's New" badge on the help/notifications area. Implementation:

- Badge appears when `last_seen_changelog_version < current_changelog_version` (stored in `user_preferences` JSONB column).
- Modal shows 2--3 most impactful changes with benefit-focused copy ("You can now do X" not "We shipped Y").
- Each change optionally links to a 2--3 step micro-tour for the new feature.
- "Don't show me updates" preference available in Settings (stored in `user_preferences`).
- Cadence: major releases (quarterly) get a modal; minor updates get a badge-only changelog entry.

**Role Change Detection:** When a user's role changes, the NATS event `TENANT_{id}.member.role_changed` is published (Section 8). A subscriber evaluates whether the new role grants access to modules the old role did not have. If so, the relevant module tour's `tour_progress` record is reset to `not_started` (only if the user had NOT previously dismissed that tour 3+ times). Example: a Viewer promoted to Manager gets the BOS tour re-triggered on their next `/bos` visit.

#### 13l.9 Edge Cases


| Scenario | Behavior | |


| **Role changed after onboarding** | Do NOT re-show the welcome screen. Show a "Your permissions have been updated" toast notification on next login. If the new role grants access to new modules, reset relevant module tour progress (see 13l.6 Everboarding). | |

| **Checklist dismissed early** | Checklist hidden from sidebar. Re-accessible via Settings. Auto-completed items continue tracking silently (for analytics). | |

#### 13l.10 Routing Logic


**Post-login onboarding routing (Server Component --- NOT middleware):**


**Important:** This check MUST NOT run in `middleware.ts`. Querying `tenant_members.onboarding_completed` requires a database call, which adds ~150ms blocking latency and kills Partial Prerendering (PPR). Middleware should only check cookie presence and JWT decode (no DB). Instead, implement this as a Server Component check behind a `<Suspense>` boundary in the authenticated layout.


- The authenticated layout's Server Component reads `tenant_members.onboarding_completed` from Supabase and calls `redirect('/welcome')` if `false` AND user was created via invitation (not org owner).

- If `onboarding_completed = true` -> render the layout children (e.g., `/dashboard`) as normal.

- The `/welcome` route's Server Component checks `onboarding_completed`; if already `true`, redirects to `/dashboard`.

- PPR sends the static shell (nav, sidebar skeleton) immediately while the onboarding check resolves behind a `<Suspense>` boundary.

```typescript
  entityType: 'submittal' | 'change_order' | 'milestone' | 'general';
  entityId?: string;                     // Links thread to a domain entity
  replyCount: number;
  participants: string[];                // User IDs
  lastReplyAt: string;                   // ISO 8601
}
```


#### Features


- **Read receipts:** `readBy[]` array per message, batch-updated every 2 seconds to reduce write load.
- **Threading:** Messages can have replies via `threadId`. Thread panel slides in from the right side of the message area.
- **Multi-platform mirroring:** Slack channel linked to Teams channel linked to Kiptra channel. Messages flow bidirectionally across all connected platforms.
- **Message reactions:** Emoji reactions with user tracking. Reactions sync across platforms (Slack emoji -> Kiptra, Kiptra emoji -> Slack/Teams).
- **System events:** 12 auto-generated event types (see `SystemMessage.systemAction`). System messages are generated by NATS event consumers when domain events occur.
- **Integration status tracking:** Per-platform connection status (`connected` | `disconnected` | `pending`) displayed in channel header.
- **File attachments:** Inline preview for images, PDFs, and CAD files. Files uploaded to Supabase Storage with signed URLs. Max file size: 25MB (matches Slack free tier limit).
- **Typing indicators:** Debounced at 500ms. Broadcast via Socket.io to channel members only.
- **Channel search:** Full-text search across all channels the user has access to. Powered by PostgreSQL `tsvector` on channel name + topic.
- **DM and group DM support:** Direct messages between 2 users. Group DMs for 3--8 users. Both use the same `ChatChannel` model with `type: 'dm' | 'group_dm'`.


#### Construction-Specific Features


- **Project-linked channels:** `projectId` on `ChatChannel` links a channel to a project. Creating a project auto-creates a corresponding chat channel. Channel inherits project team membership.
- **Submittal message type:** Submittal approval/rejection workflows generate system messages and auto-create threads. Thread `entityType: 'submittal'` links to the submittal record for context.
- **Change order notifications:** Change order creation triggers a system message in the project channel. Auto-creates a thread for discussion. Approval/rejection generates follow-up system messages.
- **Milestone completion notifications:** Milestone status changes generate system messages in the project channel.
- **Estimate attachment sharing:** Estimates can be shared as file attachments with inline preview. System message generated when an estimate is attached to a channel.
- **Delivery scheduling notifications:** Delivery date changes generate system messages with the new date prominently displayed.


#### Integration Architecture


- **Slack:** Bot token (`xoxb-*`) + Events API (`/api/v1/webhooks/slack/events`) for bidirectional sync. Scopes: `channels:history`, `channels:read`, `chat:write`, `users:read`. Events subscribed: `message`, `reaction_added`, `reaction_removed`, `member_joined_channel`, `member_left_channel`.
- **Teams:** Bot Framework SDK + Adaptive Cards for rich message rendering. Messages from Kiptra render as Adaptive Cards in Teams with action buttons. Teams messages are received via Bot Framework webhook and relayed to Kiptra via NATS.
- **Kiptra:** Native Socket.io WebSocket connection. Messages emitted to channel-specific rooms (`chat:channel:{channelId}`). Typing indicators emitted to the same room.
- **Cross-platform deduplication:** Each message has a `source` field indicating the originating platform. When a message arrives from Slack that was originally sent from Kiptra (mirrored back), the `source` field prevents re-mirroring. Dedup key: `(source, externalMessageId)`.


#### RBAC


All authenticated roles can access chat. Channel membership is separate from platform roles --- managed per-channel by channel owner/admin. Channel owners can add/remove members, mute channels, and archive channels. DMs are unrestricted between any two authenticated users within the same tenant.


#### Layout


- **Desktop (>=1024px):** Three-panel layout. Left panel: channel list (280px fixed width, collapsible). Center panel: message area (flex, fills remaining space). Right panel: thread view (400px, slides in when a thread is opened).
- **Mobile (<1024px):** Single-panel with navigation. Channel list is the default view. Tapping a channel navigates to message view. Thread view is full-screen overlay.
- **Channel list panel:** Search bar at top, section dividers (Channels, Direct Messages), unread indicators, muted channels dimmed.
- **Message area:** Message input at bottom with attachment button, emoji picker, and mention autocomplete (`@` trigger). Messages grouped by date with sticky date headers.


#### Performance Notes


- Messages paginated at 50 per load with infinite scroll (older messages load on scroll-up).
- Channel list sorted by `lastActivityAt` descending.
- Typing indicators debounced at 500ms, auto-expire after 5 seconds of no input.
- Read receipts batch-updated every 2 seconds (collected client-side, flushed to server).
- Socket.io rooms scoped to channels --- users only receive events for channels they are members of.
- Message search uses PostgreSQL full-text search (`tsvector` on `content` column) with GIN index.

#### Agent Integration


- **Leads Agent:** Auto-creates contacts from inbound emails (via SendGrid Inbound Parse). Sets `status: 'lead'`, `type: 'homeowner'` (default, adjustable). Logs `email` activity. Creates deal at `new_lead` stage if lead score >= warm threshold.
- **Sales Agent:** Creates deals and moves them through pipeline stages based on interaction outcomes. Updates `probability` based on stage-specific defaults (configurable per tenant). Logs `call`, `meeting`, and `design_presentation` activities.
- **Bookkeeping Agent:** Updates `invoiceStatus` and `arAging` from QuickBooks sync. Updates `billedToDate` and `contractAmount` from invoice data. Logs financial activities.
- **Project Management Agent:** Links deals to projects when stage reaches `contract_signed`. Updates deal stage to `pre_construction` when project is created. Logs `site_visit` and `inspection` activities.
- **Activity log captures all agent actions automatically:** Every agent execution that touches a contact or deal generates an `Activity` record with `description` prefixed by the agent name (e.g., "Leads Agent: Scored lead as hot, scheduled follow-up email").
