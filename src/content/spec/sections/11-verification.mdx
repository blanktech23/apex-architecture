### 11.11 Acceptance Criteria (Given/When/Then)


BDD-style acceptance criteria for all critical user flows. These scenarios serve as the source of truth for integration test implementation and manual QA verification. Each scenario maps to a specific verification item (V-xx) or feature area defined above.


#### Agent Execution (Trigger → Result)


Gherkin
Feature: Agent Execution

Scenario: Successful user-triggered agent execution
 Given a tenant &quot;Slate&quot; with an active Professional subscription
 And the &quot;Sales Agent&quot; agent is enabled for this tenant
 And the tenant's monthly AI spend is below 100% of their limit
 When a user with role &quot;Manager&quot; clicks &quot;Run Estimate&quot; for project &quot;Kitchen Reno&quot;
 Then a BullMQ job is queued with tenant_id, agent_type, and project context
 And an agent_executions record is created with status &quot;QUEUED&quot;
 And status transitions to &quot;RUNNING&quot; when the worker picks up the job
 And the agent executes with stopWhen: stepCountIs(17) and max_tokens: 8000
 And on success, status transitions to &quot;COMPLETED&quot; with output stored
 And a NATS event is published on &quot;TENANT_{id}.agent.sales-agent.completed&quot;
 And the user receives a real-time update via Socket.io

Scenario: Agent execution blocked by spending limit
 Given a tenant whose monthly AI spend is >= 100% of their plan limit
 When any user triggers an agent execution
 Then the API returns HTTP 429 with error code &quot;AGENT_BUDGET_EXCEEDED&quot;
 And no BullMQ job is queued
 And the dashboard shows &quot;Monthly AI budget reached — upgrade your plan&quot;

Scenario: Agent execution timeout
 Given an agent execution that has been running for 300 seconds
 When the wall-clock timeout fires
 Then the execution status is set to &quot;TIMED_OUT&quot;
 And the BullMQ job is moved to the dead letter queue
 And the user is notified via Socket.io with error code &quot;AGENT_EXECUTION_TIMEOUT&quot;
 And the event is logged to Sentry with the agent_execution_id
```

### Specification Contradiction Resolutions


Two schema conflicts identified during the DDL formalization process, resolved with rationale.


#### notification_preferences Schema Conflict (Resolved)


Resolution: Adopt Section 13 Schema (Denormalized Columns-per-Channel)


**Conflict:** Section 5 defined `notification_preferences` with a normalized row-per-channel model (`user_id, category, channel, enabled`), while Section 13 used a denormalized columns-per-channel model (`user_id, notification_type, in_app, email, push, digest_frequency`).


**Resolution:** Adopt the Section 13 schema. Rationale: (1) Fewer rows (1 per user per type vs 3), (2) Single query returns all channel preferences, (3) Easier application code, (4) `digest_frequency` column needed for batched email digests, (5) Matches better-auth notification patterns.


#### ORM Recommendation: Drizzle ORM


Recommendation: Drizzle ORM (ADR Candidate)


**Rationale:** (1) First-party better-auth adapter, (2) Type-safe queries without codegen, (3) SQL-first philosophy --- critical for RLS debugging, (4) Plain SQL migrations compatible with Expand & Contract pattern, (5) Native pgvector column support, (6) ~7x smaller bundle than Prisma, (7) No query engine overhead.


**Integration:** `drizzle-kit introspect` generates schema from existing Supabase database. `withTenantContext()` wrapper pattern preserved via Drizzle transactions with `set_config('app.tenant_id', tenantId, true)`.
