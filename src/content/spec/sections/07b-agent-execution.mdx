### Request-Response (User-Triggered)


Flow
User clicks "Run Estimate" -> API -> BullMQ job -> Agent executes
-> Stores result in Supabase -> NATS event -> WebSocket push to dashboard
```


### Scheduled (Cron via BullMQ)


Flow
BullMQ repeatable job (7 AM daily) -> CEO Agent agent
-> Pulls from all operational agent outputs -> Generates morning briefing draft
-> Stores in `drafts` table -> WebSocket notification to CEO
```


**Hybrid active-active note:** BullMQ repeatable jobs use Valkey atomic operations (ZADD with NX for delayed job scheduling) to deduplicate across nodes --- the repeatable job key includes the schedule pattern and next execution time, so only one instance is ever created per interval regardless of how many workers poll. No distributed lock protocol (e.g., Redlock) is needed; deduplication is built into BullMQ's core scheduling logic. **Jitter scheduling:** Each tenant's daily briefing is scheduled at the configured time + random(0, 30 minutes) jitter to prevent thundering herd. At 1,000 tenants, this spreads jobs over 30 minutes instead of a single spike, staying within provider rate limits.


### Event-Driven (Integration Webhooks)


Flow
JobTread webhook -> LB -> VPS endpoint -> NATS event -> Project Management Agent
-> Processes schedule change -> Flags drift -> Escalation if threshold hit
```


Inter-Agent Coordination (AI-15)


Standard event payload schema: `{ event_type, tenant_id, source_agent, target_agent, payload: JSONB, context_ref: agent_execution_id }`. The `context_ref` allows receiving agents to look up the originating execution's output without re-fetching source data. This enables the loosely-coupled coordination described in the event-driven pattern (e.g., Project Management Agent detecting schedule drift triggers Sales Agent with context about the specific drift).


### Human-in-the-Loop


Flow
Agent generates draft (email, PO, estimate) -> Stored in `drafts`
-> Dashboard shows pending approval -> User approves/rejects
-> If approved: agent executes final action (send email, create record)
```


Agent Execution State Machine (AI-19)


The `agent_executions.status` column supports 10 explicit states. All transitions are enforced by the executor framework --- no ad-hoc status strings.


State Machine
QUEUED -> RUNNING -> COMPLETED -- success
QUEUED -> RUNNING -> FAILED -- error/exception
QUEUED -> RUNNING -> TIMED_OUT -- exceeded 300s wall-clock
QUEUED -> RUNNING -> AWAITING_APPROVAL -- human-in-the-loop draft created
AWAITING_APPROVAL -> COMPLETED -- approved and executed
AWAITING_APPROVAL -> REJECTED -- human rejected
QUEUED -> CANCELLED -- removed from queue before execution
Any state -> DEAD_LETTER -- moved to DLQ after max retries
```


**Column definition:** `status TEXT NOT NULL DEFAULT 'QUEUED' CHECK (status IN ('QUEUED', 'RUNNING', 'COMPLETED', 'FAILED', 'TIMED_OUT', 'AWAITING_APPROVAL', 'REJECTED', 'CANCELLED', 'DEAD_LETTER'))`. Indexed for dashboard queries: `CREATE INDEX idx_agent_executions_status ON agent_executions (tenant_id, status)`.


**Stuck execution recovery procedure:** (1) Query `agent_executions WHERE status = 'RUNNING' AND updated_at ` (all events for one agent), `TENANT_{id}.agent.*.tool.*` (all tool events).


**NATS -> Socket.io bridge:** One NATS subscription per active tenant (`TENANT_{id}.agent.>`), multiplexed to Socket.io rooms:


- `tenant:{id}:agent:{name}` --- receives ALL events (agent detail views)

- `tenant:{id}:dashboard` --- receives `execution.*` events only (status badges)

- `tenant:{id}:escalations` --- receives `approval.*` events only (escalation queue)


**Persistence strategy:** Only `execution.start` and `execution.end` events use JetStream persistence (for reconnect replay). Turn/message/tool events use core NATS pub/sub (no persistence) since they are transient UI updates. On Socket.io reconnect, client sends `lastEventTimestamp` and server replays persisted events newer than that timestamp.

> **Spec Amendment (Attack Audit 2026-03-22):** NATS JetStream MUST be configured with `sync_interval: always` for the agent execution stream. Default `sync_interval` is 2 minutes --- on OS crash (not process crash), acknowledged-but-unsynced writes are lost. Accepts ~50% throughput reduction, which is acceptable since agent events are tens/min, not thousands/sec. This is especially critical because NATS runs standalone (no R=3 quorum for replication redundancy). Without `sync_interval: always`, a primary VPS OS crash can silently lose agent execution events that were already acknowledged to publishers.


**Selective streaming:** `message.delta` events are emitted only for chat-type agents (Leads Agent, Support Agent) where live text rendering is needed. Batch agents (Sales Agent, CEO Agent) emit only turn-level and tool-level events to avoid flooding NATS with token-level deltas. Controlled by `streamToClient: boolean` in agent config.


**Dashboard mapping:**


| Event | Dashboard Component | UI Effect | |


| `execution.start` | Agent status badge | Spinner start, "Running" indicator | |

| `execution.end` | Agent status badge + cost display | Status color change, cost summary | |

| `turn.start/end` | Step progress bar | "Step 3 of 7" with checkmarks | |

| `message.delta` | Chat panel (Support Agent) | Live text rendering | |

| `tool.start` | Tool activity indicator | "Querying JobTread API&hellip;" spinner | |

| `tool.progress` | Tool progress | "Found 12 line items&hellip;" | |

| `tool.end` | Tool result card | Success/error indicator, duration | |

| `approval.requested` | Escalation queue | New item with draft preview | |

| `approval.resolved` | Escalation queue | Item resolved, toast notification | |


**React hook:** `useAgentEvents(executionId)` subscribes to the agent's Socket.io room and maintains a local state machine (status, turnNumber, activeTools Map, completedTools array, streamingText, totalTokens, durationMs). Uses `useCallback` + `setState` with switch-case reducer pattern for each event type.


**Wave 2a deliverable:** Define the `KiptraAgentEvent` union type in `packages/shared/src/events/`, implement the NATS -> Socket.io bridge, and wire the `useAgentEvents` hook into the agent dashboard shell. The event protocol is designed before the first agent is built (Wave 3a) so all agents emit consistent events from day one.

> **Spec Amendment (Attack Audit 2026-03-22):** Consider `@socket.io/redis-emitter` as fallback for worker-to-client events when NATS is unavailable. NATS runs standalone on the primary VPS and is a SPOF for event-driven agent execution. During NATS outages, BullMQ workers can still complete jobs (via managed Valkey) but cannot relay real-time events to clients. With `@socket.io/redis-emitter`, workers publish completion/error events directly via Valkey pub/sub (which the Socket.io Redis adapter already subscribes to), providing real-time delivery resilience without NATS. Implementation: check NATS connection status before publishing; if disconnected, fall back to Redis Emitter for critical events (`execution.end`, `error`).


LLM Failure Handling (AI-07)


(a) LLM failure mid-execution -> the `executeWithEscalation()` wrapper (AI-26) automatically retries at the next model tier before aborting. If escalation is exhausted or the agent has already fired tool calls with side effects, abort the run, mark as FAILED in `agent_executions`, publish NATS error event, dead-letter to BullMQ DLQ. Do not automatically retry multi-step agents that have already fired tool calls with side effects. (b) Cross-provider fallback (Anthropic -> OpenAI) handled by both Vercel AI Gateway same-tier fallbacks and the model escalation system (cross-tier). Single-step stateless calls (classification, extraction) may fallback freely; multi-step agents mid-run only escalate if no side effects have fired. (c) `max_tokens` set per agent type based on measured output distributions: Leads Agent = 2K, Sales Agent = 8K, CEO Agent = 4K. (d) **`onError` callback mandatory:** Every `streamText` call MUST include an `onError` callback. Without it, streaming errors are silently swallowed (by design in AI SDK). Route errors to Sentry + Pino/Loki.

> **Spec Amendment (Attack Audit 2026-03-22):** Explicitly set BullMQ `attempts: 2` for agent jobs (1 retry for infrastructure failures). `executeWithEscalation()` handles application-level retries (model fallback). Throw `UnrecoverableError` for non-transient errors (400, 401, 403) to skip remaining BullMQ attempts. Verify and document Vercel AI Gateway retry behavior --- if the gateway retries internally, the total retry chain is: Vercel AI Gateway retries x BullMQ `attempts`. Only ONE layer should retry to avoid multiplicative retry storms.

> **Spec Amendment (Attack Audit 2026-03-22):** Add concrete `onError` code pattern for all `streamText` calls: `streamText({ ..., onError({ error }) { nats.publish(streamSubject, encode({ type: 'error', error: String(error) })); sentry.captureException(error); } })`. Add `error` as an explicit event type in the 13-event streaming protocol (AI-25) so clients receive structured error events via Socket.io, not just silent stream termination.


Context Assembly Strategy (AI-08)


**Concrete token budget (Sales Agent --- most context-heavy agent):**


- System prompt (cacheable prefix): ~2,000 tokens

- Tenant-specific context (company info, settings): ~500 tokens

- Retrieved documents (pgvector, top 10 chunks x 512 tokens): ~5,000 tokens

- Project data (from integrations): ~2,000 tokens

- Conversation history (if interactive): ~3,000 tokens

- Tool call results (accumulated across `stopWhen` steps): ~5,000 tokens

- **Total budget: ~17,500 tokens input** --- well within Claude Sonnet's 200K context window


**Safety rule:** `context_budget` per agent type caps total input at 50K tokens (25% of context window), leaving room for output generation and safety margin. Agents exceeding their budget trigger context truncation (oldest conversation turns dropped first) before the LLM call is made.


Daily-briefing agents fetch the last 24 hours of `agent_executions` outputs, summarized to a fixed budget (2K tokens max via Haiku) before injection as context. Conversation agents (Leads Agent) store threads in a `conversation_turns` table scoped to `(tenant_id, lead_id)` with sliding window truncation at 80% of model context limit. A `context_budget` parameter per agent type prevents unbounded context growth.

> **Spec Amendment (Attack Audit 2026-03-22):** Add per-execution cumulative token budget via `onStepFinish`: track `totalTokens += usage.totalTokens` across all steps within a single execution. Define per-agent execution budgets (e.g., Leads Agent = 30K, Sales Agent = 100K, CEO Agent = 50K). Throw `TokenBudgetExceededError` (extends `UnrecoverableError`) when exceeded. This is distinct from: (a) `max_tokens` which caps output per LLM call, (b) `context_budget` which caps input assembly, and (c) per-tenant monthly spend limits which cap billing. Without per-execution budgets, a single agent loop can burn unlimited tokens within a single run.


Execution Timeouts (AI-09)


Per-LLM-call timeout: 60s (Nano/Haiku), 120s (Sonnet) via AbortController. Per-agent-execution timeout: 300s total wall-clock. BullMQ job timeout: 360s (execution + cleanup margin). **BullMQ lock management (mandatory):** Set `lockDuration: 300_000` (5 min) and `stalledInterval: 600_000` (10 min) in Worker constructor. Add `job.extendLock()` calls in `onStepFinish` callbacks. Default 30s lock causes duplicate execution with Sonnet workloads. On timeout: mark execution FAILED, dead-letter to DLQ, notify user via Socket.io.

> **Spec Amendment (Attack Audit 2026-03-22):** Reduce `lockDuration` from 300_000 to 180_000, `stalledInterval` from 600_000 to 300_000. Shorter lock + active extension recovers faster from crashed workers (crashed worker holds lock for full lockDuration, blocking recovery). Add initial `job.extendLock(job.token!, modelTimeout + 30_000)` BEFORE the first `generateText`/`streamText` call (not just in `onStepFinish`) --- the first LLM call can take 60-120s, during which no `onStepFinish` fires. Add `ResettableTimeout` AbortController per model (reset on each `onStepFinish`) to catch hung LLM calls without killing legitimately long multi-step runs. Worker config: `lockDuration: 180_000, stalledInterval: 300_000, maxStalledCount: 2`.

> **Spec Amendment (Attack Audit 2026-03-22):** Add `maxStalledCount: 2` to worker config. Mandate `worker.on('stalled', (jobId) => logger.warn('Job stalled', { jobId }))` and `worker.on('error', (err) => logger.error('Worker error', { err }))` event handlers --- workers fail silently without these. Add health check watchdog: if no jobs processed within 5 min and queue depth > 0, auto-restart worker (mitigates ioredis #610 BZPOPMIN reconnection bug).

> **Spec Amendment (Attack Audit 2026-03-22):** Document and verify Vercel AI Gateway timeout behavior. Confirm AI SDK AbortController timeout is the binding constraint (not a hidden gateway timeout). Add explicit verification during Wave 2a setup --- test that AbortController at 120s actually aborts the call, not just the client-side wait.


Conversation Thread Lifecycle


Threads with no activity for 90 days are archived to cold storage. Full history retained for audit/compliance. Estimated storage: ~10MB/month/tenant at 100 leads/month.


### Timezone Handling


Construction contractors operate across US time zones. Correct timezone handling is critical for scheduled agent executions (daily briefings, cron jobs) and customer-facing timestamps.


- **`timezone` column on `tenants` table:** Stores IANA timezone identifier (e.g., `America/New_York`, `America/Los_Angeles`). Set during onboarding --- auto-detected from browser with manual override. Already present in the tenants schema (Section 5).

- **UTC storage:** All `created_at` / `updated_at` timestamps stored in UTC across Supabase, Valkey, and NATS. No local time in the database --- ever.

- **BullMQ scheduled jobs:** Daily briefings, report generation, and all cron-based agent triggers use the tenant's configured timezone for scheduling. The BullMQ repeatable job pattern converts tenant local time to UTC for Valkey storage. Jitter scheduling (Section 7b) applies after timezone conversion.

- **UI display:** All timestamps displayed in the tenant's local timezone via client-side conversion using `Intl.DateTimeFormat` with the tenant's IANA timezone. No server-side timezone formatting.

- **Wave 1 deliverable:** Add timezone selection to onboarding flow and tenants schema. Timezone is a required field --- no default assumed.


### Agent System Prompts (Complete)


Production-ready system prompts for all 7 agents. Each prompt includes identity context, behavioral guidelines, tool usage rules, output format specifications, safety rails, model routing preferences, and step limits. Template variables (`{{tenant_name}}`, `{{tenant_id}}`, etc.) are injected at execution time from the tenant and user context.


**Version control:** These prompts are stored in the `agent_prompt_versions` table (platform-scoped). Each prompt has a version number and SHA-256 hash. The `agents` table references a specific `prompt_version` per tenant, enabling canary rollout (10% -> 90%) with automated rollback on quality regression.
