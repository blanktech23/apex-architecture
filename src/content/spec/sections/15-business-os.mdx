### 15.1 Overview & Strategic Rationale


The Entrepreneurial Operating System is a business management methodology used by over 130,000+ companies worldwide (per EOS Worldwide's 2024 data). It provides a structured framework for vision alignment, quarterly goal-setting, weekly meeting cadence, KPI tracking, issue resolution, and people management. EOS is *enormously* popular in construction --- Sachse Construction grew from $35M to $138M during its scaling phase using it, and the methodology's emphasis on accountability, measurables, and execution discipline maps directly to how construction companies operate.


The dominant EOS software platform, Ninety.io, has a critical limitation: **historically limited integration capabilities.** Ninety launched Scorecard Integrations in Q1 2026, is beta-testing a limited Scorecard API for Q2 2026, added a Monday.com integration (Dec 2025 beta), and has To-Dos integration with Todoist/Asana/ClickUp in beta. However, it still lacks a general-purpose API, webhook subscriptions, or construction-specific integrations, and charges $11.20--$16 per user per month. This creates a strategic opening for Kiptra AI.


**Key insight:** No construction project management tool offers business operating system features. No EOS tool offers construction PM features. Kiptra AI is uniquely positioned to deliver both in a single platform --- with AI augmentation, automated KPI data, and an open API that Ninety.io cannot match.

**Persistence:** Machine state is serialized to a `workflow_state` JSONB column on the parent record. On server restart or reconnection, the machine is rehydrated from persisted state. BullMQ handles delayed transitions (timer-based section advances) with durable job scheduling.


Architecture Decision: XState v5 Hybrid Execution (Server-Authoritative Workflows)


**Decision:** All five named state machines (Meeting Lifecycle, Quarterly Goal Lifecycle, Review Workflow, Issue Resolution, Approval Chain) execute on the server (Hono API). Client-side execution is limited to ephemeral UI state (section countdown timers, panel toggles, drag state).


**Why not client-only:** Meetings have multiple concurrent users driving the same machine. Browser tabs close mid-meeting. Browser timers are throttled in background tabs. Client-side execution creates split-brain risk with no single source of truth.


**Why not server-only:** Timer countdowns and UI micro-state would require API round-trips for every tick, adding latency that degrades UX.


**Architecture:**


- **Event flow:** Client sends events via `POST /api/{entity}/{id}/events`. Hono routes to XState machine. Machine transitions, persists new `workflow_state` JSONB, schedules BullMQ delayed jobs for timeouts.

- **Broadcast:** New state broadcast to all connected clients via Supabase Realtime Broadcast channel. Clients render server state as source of truth.

- **Machine registry:** LRU cache of active machine instances in Hono process. Machines loaded from DB on first event, evicted after 30 minutes of inactivity.

- **Horizontal scaling:** Machine events routed through NATS JetStream (`machine.{type}.{id}`) with single consumer per machine ID. Prevents concurrent processing across 2 VPS nodes.

- **Meeting timers:** Server schedules BullMQ delayed job per section duration. Job fires `SECTION_TIMEOUT` event to machine via NATS. Durable across server restarts (Valkey-backed). Client shows local countdown from `section_started_at + duration_minutes`, synced on reconnect.

- **Reconnection:** Client fetches current `workflow_state` from API on reconnect. No client-side machine rehydration --- client is a pure state renderer.


#### 15.4.3 Analytics & Visualization


The Analytics Dashboard and KPI Dashboard require rich, interactive data visualization with drill-down capabilities, cross-filtering, and responsive layouts.


- **Chart library:** Recharts for all charts (line, bar, area, combo, sparklines) + shadcn/ui Card components for KPI card layouts

- **Data layer:** Postgres materialized views for pre-computed aggregates (goal completion rates, KPI trends, meeting effectiveness). Refreshed via BullMQ cron jobs (every 15 minutes for active data, hourly for historical).

- **Drill-down:** Click any aggregate to see constituent records. KPI off-track -> show weekly values -> show individual data points. Goal completion rate -> show individual goals -> show milestones.

- **Cross-filtering:** Select a team in one widget, all other widgets filter to that team. Time range selector affects all charts simultaneously.

#### 15.4.4 Calendar Sync


Meeting Hub requires two-way calendar synchronization so that Weekly Syncs, Quarterly Reviews, and ad-hoc meetings appear on users' Google Calendar and/or Microsoft Outlook calendars.


- **Google Calendar:** Google Calendar API v3 via OAuth2. Scopes: `calendar.events` (read/write). Incremental sync via `syncToken` for efficient polling.

- **Microsoft Outlook:** Microsoft Graph Calendar API via MSAL (Microsoft Authentication Library). Delta queries for incremental sync.

- **Recurring events:** RFC 5545 RRULE support for recurring meetings (weekly syncs are always recurring). Store RRULE in DB; generate occurrences on read.

- **Sync direction:** Two-way. Meeting created in Kiptra -> event created in external calendar. Meeting rescheduled in external calendar -> Kiptra meeting updated. Conflict resolution: Kiptra is source of truth for meeting *content*; external calendar is source of truth for *time* changes.

- **Sync jobs:** BullMQ repeatable jobs per user/calendar connection. Poll interval: 5 minutes. Webhook-based push notifications where supported (Google Push Notifications, Microsoft Graph subscriptions).


#### 15.4.5 Notification System


Business OS modules generate a high volume of notifications: meeting reminders, action item due dates, KPI off-track alerts, goal status changes, review assignments, and issue escalations. The notification system must be channel-agnostic, user-configurable, and deduplication-aware.


- **Event-driven architecture:** All Business OS events publish to NATS JetStream. Notification service subscribes to relevant subjects and generates notification records.

- **Channel router:** Each notification is routed to one or more channels based on user preferences:


- **In-app:** Socket.io push to connected clients. Stored in `notifications` table with read/unread status.

- **Email:** SendGrid transactional emails with customizable templates. Batched digest option (daily/weekly).

- **Push:** Firebase Cloud Messaging (FCM) for mobile/desktop push notifications.


- **User preferences:** Per-notification-type, per-channel toggles. Example: "KPI off-track" -> in-app + email; "Meeting starting" -> in-app + push; "Announcement posted" -> in-app only.

- **Deduplication:** Hash-based dedup within a 5-minute window. Prevents duplicate notifications from retry logic or event replays.

- **Rate limiting:** Maximum 10 notifications per user per hour per channel. Excess notifications batched into a digest.

- **Digest scheduling:** Users can opt into daily or weekly email digests instead of individual notifications. BullMQ cron job aggregates and sends.


#### 15.4.6 PDF & Report Generation


Several modules require PDF export: meeting recaps, vision plan documents, quarterly goal reports, KPI dashboard snapshots, and review summaries.


- **Structured reports:** React-PDF (`@react-pdf/renderer`) for programmatic PDF generation. React components render directly to PDF with full layout control. Used for meeting recaps, goal reports, and review documents.

- **Complex visual reports:** Puppeteer for rendering HTML/CSS to PDF when charts and complex layouts are needed (KPI dashboard snapshots, analytics reports). Headless Chrome in a Docker sidecar container.

- **Async generation:** All PDF generation runs as BullMQ jobs to avoid blocking API requests. Job creates PDF, uploads to object storage (S3-compatible), and notifies user via the notification system.

- **Templates:** Meeting recap, Strategic Vision Plan export, Quarterly Goal report, KPI Dashboard snapshot, Review summary, Responsibility Map org chart.


#### 15.4.7 Knowledge Portal


The Knowledge Portal is the most content-heavy module --- a combined learning management system and internal wiki. Its editor and content model require special attention.


- **Editor:** TipTap (ProseMirror-based) with extensions for tables, images, embeds, checklists, callouts, and code blocks. Collaborative editing via `prosemirror-collab` (same infrastructure as vision plan text).

- **Content model:** Articles are composed of `ContentBlock` records (JSONB) supporting text, image, video, embed, checklist, and quiz block types. Blocks are ordered and independently versioned.

- **Search:** Postgres `tsvector` for full-text search across article content + `pg_trgm` for fuzzy/partial matching. Search results ranked by relevance with snippet highlighting.

- **Assignment & tracking:** Seat-based assignment (assign articles/collections to seats, not people --- consistent with Responsibility Map). Learning progress tracked per-user-per-article with completion percentage, quiz scores, and time spent.

- **Content freshness:** Configurable review cycles per article (30/60/90/180 days). Automated notifications to content owners when review is due. Stale content flagged in UI with amber indicators.


#### 15.4.8 Organizational Data Modeling


Business OS modules fundamentally depend on organizational hierarchy: teams, seats (roles), people, and reporting relationships. This data model underlies the Responsibility Map and scopes data access throughout all modules.


- **Hierarchy storage:** Postgres `ltree` extension for efficient hierarchical queries (ancestors, descendants, subtree operations). Alternative: closure table if `ltree` proves limiting for complex org structures. Decision: start with `ltree`, migrate to closure table only if needed.

- **Seat-person separation:** Seats (roles/functions) are distinct from people. A seat can be vacant, held by one person, or shared. A person can hold multiple seats. This is core EOS methodology and must be strictly modeled.

- **Access control:** RBAC (Role-Based Access Control) layered on top of Supabase RLS. Team-scoped data: users only see data for teams they belong to. Seat-scoped permissions: certain actions require specific seat assignments (e.g., only the meeting facilitator can advance sections).

- **Audit trail:** Trigger-based audit log on all Business OS tables. Records who changed what, when, and the before/after values. Required for quarterly reviews and compliance.

- **Polymorphic relationships:** `ItemLink` table for cross-module references (e.g., linking an issue to a quarterly goal, linking an action item to a meeting section). Uses `source_type`/`source_id` + `target_type`/`target_id` pattern.

#### AI Pre-Population Mechanics


The AI Assistant Agent does not use live LLM calls for suggestion generation during onboarding --- this would add latency and unpredictability to a critical first-impression flow. Instead:


- **Static templates:** The `onboarding_templates` table stores curated suggestions per industry, loaded on wizard initialization. Zero latency, deterministic output.

- **AI refinement (optional):** If the user clicks "Customize with AI" on any step, the AI Assistant Agent activates with a conversational interface: "Tell me about your business and I'll suggest better KPIs." This uses the standard AI gateway (Section 4) with Claude Haiku for fast responses (~500ms).

- **Learning from edits:** When users modify AI suggestions, the deltas are logged to `onboarding_analytics` for future template refinement. If 80%+ of GCs change "Continuous Improvement" to "Never Settle," the template updates.


TypeScript
// Onboarding template lookup — zero LLM latency
interface OnboardingTemplate {
 industry: IndustryVertical;
 companySize: CompanySizeRange;
 coreValues: string[];
 suggestedSeats: { title: string; seatType: SeatType; responsibilities: string[] }[];
 suggestedKPIs: { title: string; unit: KPIUnit; goalDirection: GoalDirection; goalValue: number | null }[];
 meetingRecommendation: 'compact' | 'standard' | 'extended';
}

async function getOnboardingTemplate(
 industry: IndustryVertical,
 companySize: CompanySizeRange
): Promise {
 const template = await db
 .selectFrom('onboarding_templates')
 .where('industry', '=', industry)
 .where('company_size', '=', companySize)
 .executeTakeFirst();

 if (!template) {
 // Fallback to "Other / Custom" template
 return getOnboardingTemplate('other', companySize);
 }
 return template;
}

**Decisions locked:**


**1. Wizard-first, not dashboard-first.** New users see the wizard immediately after account creation, not the main dashboard. Rationale: an empty dashboard with zero data is worse than a guided setup flow.


**2. Quarterly goals deferred.** Goals are excluded from the minimum viable setup. Users create goals after their first meeting, when they understand the system. This reduces time-to-first-meeting by ~15--20 minutes.


**3. Static templates over live AI.** Onboarding suggestions come from curated templates, not live LLM calls. Rationale: predictability and zero-latency matter more than personalization during first-run UX. Live AI is available as an opt-in refinement.


**4. 6-step minimum, not 18-module full setup.** Only 6 of 18 modules require configuration before first use. The remaining 12 modules are introduced progressively over 4 weeks via the sidebar checklist.

#### Error and Retry Paths


| Failure Scenario | Detection | Recovery Path | User Impact | |


|
**KPI data source unavailable** (external API --- QuickBooks, CRM) |
KPI/Scorecard Agent `ingestExternalData` tool returns connection error or timeout (>30s) |
BullMQ retry with exponential backoff (3 attempts, 5min/15min/60min). After 3 failures: mark measurable as `source: &apos;stale&apos;`, NATS publish `bos.kpi.data_source_failed`, create notification for KPI owner: &quot;[Source] data unavailable --- enter value manually or retry.&quot; Manual entry fallback always available. |
KPI shows last known value with amber &quot;stale&quot; indicator. Owner can enter manually. Meeting proceeds with stale data flagged. |
 |

|
**Meeting abandoned mid-session** (facilitator disconnects, browser crash) |
XState machine in `in_progress` state with no `section_advanced` event for >15 minutes. Socket.io disconnect event for facilitator. BullMQ health check queries `meetings WHERE workflow_state->>&apos;value&apos; = &apos;in_progress&apos; AND updated_at 7 days past due)** |
Daily BullMQ cron job (8 AM tenant-local) queries `action_items WHERE completed_at IS NULL AND due_date `parent_seat_id` hierarchy). Day 7+ overdue: auto-create issue in next meeting&apos;s Issue Resolution section (title: &quot;Overdue action item: [title] (7+ days)&quot;, category: &apos;accountability&apos;). At next meeting: incomplete items carry forward with `carried_from_id` reference. Carry count tracked for accountability reporting. |
Owner receives escalating notifications. Item persists across meetings until resolved. Carry count visible in meeting recap and analytics. |
 |

|
**LLM failure during recap generation** |
`generateRecap` Haiku call returns error or timeout (>60s) |
Retry once with same context. If second failure: store meeting data without recap (`recap_html = NULL`), mark meeting as `completed_no_recap`. Queue BullMQ job to retry recap generation in 30 minutes. NATS publish `bos.meeting.recap_failed`. Notification: &quot;Meeting completed. AI recap will be available shortly.&quot; |
Meeting concludes normally. Recap arrives late (email delayed). All action items and resolutions captured regardless of recap status. |
 |

|
**Materialized view refresh failure** |
BullMQ job for `REFRESH MATERIALIZED VIEW CONCURRENTLY` fails (lock contention, OOM) |
Retry in 5 minutes. Dashboard falls back to Valkey-cached stale data (TTL extended from 5min to 30min on failure). Alert to operations channel. Views remain queryable with stale data --- `CONCURRENTLY` ensures the old snapshot is served during refresh. |
Dashboard shows slightly stale data (up to 30 min old). No user-visible error. |
 |


**Key insight:** The weekly loop is resilient by design --- every failure mode has a manual fallback that preserves the meeting cadence. KPI data can be entered manually, meetings can be paused and resumed, action items carry forward across weeks, and recaps can be regenerated asynchronously. The methodology works even when the AI layer is degraded.


### 15.5 Database Schema


The Business Operating System adds approximately 30+ new tables to the Kiptra database, organized by domain. All tables follow Kiptra conventions: UUID primary keys, `tenant_id` for multi-tenancy, `created_at`/`updated_at` timestamps, and soft-delete via `deleted_at`.


**Key insight:** All Business OS tables inherit Kiptra's tenant isolation model: Row-Level Security (RLS) policies on every table, `tenant_id` column indexed and enforced, and all queries wrapped in `withTenantContext()` (see Section 3). No additional multi-tenancy work is required --- the pattern is already established.


Architecture Decision: Rename companies to tenant_eos_settings (ADR-SCHEMA-001)


**Decision: Option C --- Rename `companies` to `tenant_eos_settings`** --- a clearly scoped settings table with a 1:1 foreign key to `tenants`.


| Criterion | A: Merge into tenants | B: companies FK->tenants | C: tenant_eos_settings (chosen) | |


|
 **Schema clarity** |
 Muddies the core table --- `tenants` becomes a dumping ground for every module's config. |
 Ambiguous naming --- `companies` sounds like a parallel top-level entity. |
 **Best.** Name explicitly communicates: "this is EOS-specific configuration for a tenant." |
 |

|
 **Migration safety** |
 Requires ALTER TABLE on the high-traffic `tenants` table. |
 Clean --- new table, no core table changes. |
 **Best.** New table, zero changes to `tenants`. |
 |

|
 **Future extensibility** |
 Every new module adds more columns to `tenants` --- "God table" anti-pattern. |
 Sets a precedent but naming doesn't follow a pattern. |
 **Best.** Establishes the `tenant_{module}_settings` naming convention. K&B can add `tenant_kb_settings`. |
 |


**Key constraint:** The `tenant_eos_settings` table has a `UNIQUE` constraint on `tenant_id`, enforcing 1:1 cardinality. It is NOT a child table --- it is a settings extension. RLS policy uses the same `tenant_id` pattern as all other tables.

#### Agent Interaction Patterns


Business OS agents frequently collaborate on cross-cutting operations:


- **Meeting -> Execution:** Meeting Facilitator extracts action items and issues during recap generation, then publishes events that the Goal & Alignment Agent and KPI/Scorecard Agent consume to update relevant records.

- **KPI -> Meeting:** KPI/Scorecard Agent identifies off-track measurables and automatically adds them to the next Weekly Sync's issues section via the Meeting Facilitator Agent.

- **Review -> People:** People & Review Agent pulls goal completion data from Goal & Alignment Agent and KPI performance from KPI/Scorecard Agent to pre-populate review forms.

- **AI Assistant -> All:** AI Assistant Agent has read access to all other agents' data via tool calls, enabling conversational queries like "How did our team do this quarter?" that span multiple domains.
