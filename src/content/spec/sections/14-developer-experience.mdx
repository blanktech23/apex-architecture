This section consolidates the monorepo structure, directory layout, and developer-facing conventions that complement the infrastructure defined in Section 12. These specifications ensure a new developer can clone the repository and be productive within a single session.


### Monorepo Structure


**Tooling:** pnpm workspaces + Turborepo for task orchestration and build caching.


**Package structure:**


- **`packages/frontend/`** --- Next.js 16 App Router (deployed to Vercel)

- **`packages/backend/`** --- Hono API server + BullMQ workers + NATS subscribers (deployed to VPS via Docker)

- **`packages/shared/`** --- Zod schemas, TypeScript types, domain model interfaces (consumed by both frontend and backend as a workspace dependency)

- **`packages/config/`** --- Shared ESLint, TypeScript, and Prettier configurations

- **`packages/editor/`** --- TipTap rich-text editor configuration, extensions, and wrapper components. Powers collaborative document editing (meeting recaps, SOPs, project notes) with real-time sync via Hocuspocus (consumed by frontend)


**Build order:** `shared` -> `editor` + `backend` + `frontend` (parallel, editor and frontend co-build). TypeScript project references: each package has `composite: true`, root `tsconfig.json` has `references` array pointing to all packages. Turborepo pipeline: `build` depends on `^build` (shared builds first). The shared package exports via direct file imports (e.g., `@kiptra/shared/schemas/user`), NOT barrel exports.

### Testing Conventions


- **Unit tests:** Co-located with source files as `*.test.ts` (e.g., `src/routes/agents.ts` -> `src/routes/agents.test.ts`). Run via `vitest`.

- **Integration tests:** Located in `packages/backend/test/integration/`. These test real database connections, NATS pub/sub, BullMQ job processing, and external API contracts. Run via `vitest --project integration` with a test Docker Compose stack.

- **E2E tests:** Located in `packages/frontend/e2e/`. Playwright tests against the running application. Run via `pnpm --filter frontend test:e2e`.

- **Coverage target:** 80% line coverage minimum (enforced by CI). Coverage reports generated by Vitest and uploaded to Codecov.


### Git Workflow & Commit Conventions


- **Branching strategy:** Trunk-based development with short-lived feature branches. All branches merge to `main` via pull request. Branch naming: `feat/`, `fix/`, `chore/`, `docs/` prefixes.

- **Commit convention:** Conventional Commits enforced by commitlint. Format: `type(scope): description`. Types: `feat`, `fix`, `chore`, `docs`, `test`, `refactor`, `perf`, `ci`. Scopes: `frontend`, `backend`, `shared`, `editor`, `infra`. This enables future automated semantic versioning via release-please.

- **Pre-commit hooks:** Husky + lint-staged. On every commit: ESLint (fix mode) + Prettier (write mode) + TypeScript type-check (`tsc --noEmit`) on staged files only. Hook runs in `pnpm install` -> `cp .env.example .env.local` -> `pnpm dev` -> productive in under 1 hour.

### API Contracts & Domain Model (Complete Reference)


Single source of truth for all TypeScript types, Zod schemas, API route contracts, WebSocket events, and NATS subject catalog. This file lives at `packages/shared/src/contracts/api-contracts.ts` and is consumed by both frontend (TanStack Query hooks) and backend (Hono route handlers).


**Four parts:** (1) Canonical domain model --- Zod schemas for Tenant, User, Agent, Execution, Escalation, Draft, Integration, Support, Notification, and Billing entities. (2) API route definitions --- request/response schemas for all Hono endpoints. (3) WebSocket event catalog --- Socket.io event types for real-time dashboard updates. (4) NATS subject catalog --- JetStream subjects with publisher/consumer documentation.


| Part | Content | Key Types | |


| **Part 1** | Canonical Domain Model | `TenantSchema`, `UserSchema`, `AgentSchema`, `AgentExecutionSchema`, `EscalationSchema`, `DraftSchema`, `IntegrationSchema`, `SupportConversationSchema`, `NotificationSchema` | |

| **Part 2** | API Route Definitions (Hono + Zod) | Tenants CRUD, Agents CRUD, Executions, Escalations, Drafts, Integrations, Support, Billing, Admin, Webhooks, Onboarding | |

| **Part 3** | WebSocket Event Catalog (Socket.io) | `ServerToClientEvents`, `ClientToServerEvents` --- agent status, notifications, support chat, connection lifecycle | |

| **Part 4** | NATS Subject Catalog (JetStream) | 27 subjects across Agent, Escalation, Draft, Billing, Integration, Notification, and System domains | |

// Organized by domain. Each route specifies method, path, schemas, auth, and rate limit.

// ---------------------------------------------------------------------------
// Rate Limit Tiers (Section 5 + Section 12)
// ---------------------------------------------------------------------------

export const RateLimitTier = {
 /** Auth endpoints: 5-10 req/min/IP */
 Auth: 'auth',
 /** Agent execution: 10 req/min/tenant */
 AgentExecution: 'agent_execution',
 /** Escalation approval: 30 req/min/user */
 EscalationApproval: 'escalation_approval',
 /** Standard authenticated: 100 req/min/user */
 Standard: 'standard',
 /** Webhook ingestion: unlimited (rate limiting via provider) */
 Webhook: 'webhook',
 /** Admin operations: 30 req/min/user */
 Admin: 'admin',
 /** Data export: 1 req/24h/tenant */
 Export: 'export',
} as const;
export type RateLimitTier = (typeof RateLimitTier)[keyof typeof RateLimitTier];

/**
 * Parse a tenant room name and extract tenant ID.
 * Room format: tenant:{tenantId}:{channel}
 */
export function parseTenantRoom(roomName: string): {
 tenantId: string;
 channel: string;
} | null {
 const match = roomName.match(/^tenant:([a-f0-9-]+):(.+)$/);
 if (!match) return null;
 return { tenantId: match[1], channel: match[2] };
}

/**
 * BullMQ job ID builder — deterministic for deduplication (Section 9, Wave 2).
 * Format: ${tenantId}:${agentId}:${triggerType}:${windowId}
 */
export function buildJobId(
 tenantId: string,

 triggerType: TriggerType,
 /** Time-bucket (floor to 5-minute intervals for scheduled jobs) */
 windowId?: string,
): string {
 const window = windowId ?? Math.floor(Date.now() / 300_000).toString();
 return `${tenantId}:${agentId}:${triggerType}:${window}`;
}
