### CI/CD Pipeline (Wave 1, Day 1)

**Spec Amendment (Attack Audit 2026-03-22):** All CI workflows MUST have: `concurrency: { group: ci-${{ github.ref }}, cancel-in-progress: true }` to prevent redundant parallel runs, `timeout-minutes` on every job (10 for lint, 20 for test, 15 for deploy), and `fetch-depth: 1` on all checkout steps. These are GitHub Actions non-negotiable standards per KB --- without explicit timeouts, the default is 6 hours, and a hung job burns Actions minutes silently.


- **Every PR:** Lint (ESLint) -> type-check (tsc) -> unit tests (Vitest) -> build

- **Merge to main:** Integration tests -> deploy to staging -> smoke tests -> deploy to production

- **Frontend:** Vercel auto-deploys from main (preview deploys on PRs for free)

- **Backend:** GitHub Actions SSH deploy to both VPS nodes (secondary first, then primary), `docker compose pull && docker compose up -d`

- **Rollback discipline:** During the deploy step, after pulling and starting the new image, explicitly tag the outgoing image as 'previous' and do NOT prune it. Both the current and previous image tags must be resident on each VPS node at all times. Rollback command: `IMAGE_TAG=previous docker compose up -d --force-recreate`. Verification: after rollback, confirm `docker compose ps` shows all services healthy and `curl -f http://localhost:3000/health/live` returns 200 within 30 seconds. Rollback SLA: under 90 seconds (matching LB failover window). This requires images to be pre-pulled --- pulling a new image at rollback time adds 2--5 minutes, which is unacceptable during an incident.

- **Spec Amendment (Attack Audit 2026-03-22):** SSH keys for VPS deploy must be scoped to a deploy-only user (not root). The deploy user has permissions limited to `docker compose` commands and image pull/tag operations. Implement key rotation on a quarterly schedule. Consider `tailscale ssh` with OIDC-backed access for zero-static-credential deploys --- this eliminates stored SSH keys entirely and ties VPS access to GitHub Actions OIDC identity.

- Branch protection on `main`: require passing checks before merge

- All third-party actions pinned to full SHA. Workflow-level `permissions: {}` with per-job scoping. `persist-credentials: false` on checkout.

- **Spec Amendment (Attack Audit 2026-03-22):** Add three CI workflows: (1) CodeQL scanning on PR + weekly schedule with `security-extended` queries (catches 50%+ more vulnerabilities than default query suite). (2) `actions/dependency-review-action` on all PRs with `fail-on-severity: moderate`, `deny-licenses: GPL-3.0, AGPL-3.0` to block vulnerable or copyleft-licensed dependency additions. (3) Enable secret scanning with push protection. Add custom secret patterns for Supabase service role keys (`sbp_`), SendGrid API keys (`SG.`), and Anthropic API keys (`sk-ant-`) to catch platform-specific credential leaks that GitHub's built-in patterns may miss. Pin all security workflow actions to full SHA hashes per KB non-negotiable standards.

- **Spec Amendment (Attack Audit 2026-03-22):** SBOM generation via `anchore/sbom-action` on each release (pinned to full SHA). Produces SPDX or CycloneDX format SBOM attached as release artifact. Useful for SOC 2 prep (software inventory requirement) and enterprise customer procurement questionnaires. Additionally, add `pnpm audit --audit-level=high` as a CI gate --- PRs introducing dependencies with known high/critical vulnerabilities are blocked.

- **Dependency management:** Configure Renovate (preferred over Dependabot for monorepo support) as a Day 1 GitHub Actions item. Configuration: weekly minor/patch PRs grouped by layer (frontend, backend, infra), semantic versioning constraint pinning for known breaking-change packages (Drizzle, AI SDK, better-auth), auto-merge for patch updates that pass CI. For AI SDK specifically: all AI SDK usage must be encapsulated behind a thin adapter layer (`src/lib/ai/`) so version upgrades only require changing the adapter, not every tool definition and agent file across the codebase.


### CI Quality Gates (All Enforced --- PRs Blocked on Failure)


- **ESLint:** Zero errors (warnings allowed but tracked). Custom rules: `no-restricted-properties` for `c.notFound()`, `no-restricted-syntax` for `enum`, `no-restricted-imports` for barrel exports.

- **TypeScript:** `tsc --build --noEmit` with zero errors.

- **Vitest:** All tests pass, 80% line coverage minimum.

- **Migration compatibility:** Custom CI check that diffs the previous and current migration files against the running code to detect column removal/rename that would break the previous code version during rolling deploy.

- **CI time target:** Under 8 minutes for the full pipeline (lint + typecheck + test + build). Use Turborepo remote caching to achieve this.

- **Spec Amendment (Attack Audit 2026-03-22):** When test suite exceeds 5 minutes, implement Vitest sharding: `--shard=1/4` across 4 matrix jobs. Run full matrix on `main` branch pushes; run single shard on PR pushes for fast feedback. Use `fail-fast: true` in matrix strategy to cancel remaining shards on first failure. This matches the KB pattern of splitting fast (lint) and slow (test) jobs with `needs:` dependency for fail-fast savings.


### Dual-VPS Architecture (Day 1)


Two identical DigitalOcean CPU-Optimized droplets (4vCPU/8GB, $80/mo each = $160/mo total) behind a DigitalOcean Load Balancer ($12/mo). Intentionally over-engineered for reliability from Day 1.


- **Primary VPS:** Runs the full Docker Compose stack (Hono backend, NATS). Valkey is DigitalOcean Managed (external service, not in Docker Compose). BullMQ is a library within the Hono backend, not a separate service

- **Secondary VPS:** Hybrid active-active *for HTTP request handling only*. Both nodes serve HTTP traffic simultaneously via the Load Balancer. The secondary handles API requests and runs BullMQ workers (coordinated via managed Valkey). Shared state coordination via external managed Valkey (DigitalOcean Managed Valkey --- mandated, not optional) and NATS leaf-node topology. Receives the same Docker image deployments. Health checks every 30 seconds.

- JetStream HA Limitation:** The secondary VPS connects to the primary's NATS as a leaf node. JetStream consumers on the leaf node route all operations through the hub (primary). If VPS-1 (hub) dies, *all* JetStream processing stops --- the secondary provides zero redundancy for event-driven agent execution. Agent triggers via NATS events are primary-only until a 3rd NATS node is added to form a Raft quorum. The secondary's value is HTTP request handling and BullMQ job processing (via managed Valkey), not event-bus resilience.

- **Load Balancer:** DigitalOcean managed LB distributes traffic. If primary fails health check (3 consecutive failures = 90 seconds), traffic automatically routes to secondary.

- **Shared State:** PostgreSQL (Supabase) is fully external. Valkey runs as a single managed instance (DigitalOcean Managed Valkey) --- not per-node. Socket.io uses `@socket.io/redis-adapter` for cross-node event delivery. NATS runs in standalone mode with file-based persistence on the primary node (not clustered JetStream --- two nodes cannot form a Raft quorum); the secondary node connects to the primary's NATS as a leaf node. State divergence risk is minimal but not zero --- a network partition between nodes and the shared Valkey/NATS instances requires health-check-driven failover (covered by LB policy above).

- **Cost:** $160/mo (2x VPS) + $12/mo (LB) = $172/mo. Only $92/mo more than single VPS. Insurance against total platform outage for all customers.

- **Failure Modes:** If primary VPS fails, the secondary continues serving HTTP traffic via the Load Balancer (90-second health check failover). However, NATS JetStream (standalone on primary) becomes unavailable --- event-driven agent execution is suspended until primary recovers or manual failover promotes a new NATS primary. Valkey (managed instance) has its own availability SLA (~99.9% = ~8.7 hours/year potential downtime). During Valkey unavailability, BullMQ job processing, Socket.io cross-node delivery, and distributed cache are all degraded. Agent execution designed to be re-triggerable from BullMQ job state, treating NATS as a delivery mechanism rather than a durable store. Note: With `@socket.io/redis-adapter`, Socket.io cross-node delivery is NOT affected by NATS failure --- real-time dashboard updates continue via Valkey pub/sub independently. Database failure (Supabase) is the remaining SPOF --- mitigated by Supabase's managed HA infrastructure + daily pg_dump backups to DigitalOcean Spaces.


### Docker Compose Hardening Requirements (Wave 1)


Production Docker Compose configuration must include the following for all services. Without these, the 8GB droplet is vulnerable to disk exhaustion, OOM kills, and silent container failures.


- **Log limits (INFRA-01, HIGH):** Every service MUST have `logging: { driver: &quot;json-file&quot;, options: { max-size: &quot;10m&quot;, max-file: &quot;3&quot; } }`. Without log rotation, a verbose service fills the 8GB droplet disk and takes down all services.

- **Memory limits (INFRA-02, HIGH):** Every service MUST set `deploy.resources.limits.memory`. Node.js containers (Hono backend) MUST also set `--max-old-space-size` to 75% of the container memory limit (e.g., 1536m for a 2GB container). Without explicit limits, OOM kills are silent and undiagnosable.

- **Health check conditions (INFRA-03, HIGH):** All `depends_on` declarations MUST use `condition: service_healthy`, not just container start. Without health check conditions, the Hono backend starts before NATS is actually ready to accept connections, causing startup race conditions.

- **Named volume ownership (INFRA-04, MED):** Docker named volumes are created with root ownership (Docker bug #3270). Non-root containers (see INFRA-05) need an explicit `chown` in the entrypoint script to fix volume permissions on first run.

- **Dockerfile requirements (INFRA-05, MED):** All Dockerfiles MUST use: (a) multi-stage builds (builder + production stages to minimize image size), (b) `USER node` (non-root execution), (c) `tini` or `dumb-init` as PID 1 init process for proper signal handling. Without an init process, `docker stop` sends SIGTERM but Node.js does not forward it to child processes --- after 10s Docker sends SIGKILL, which can cause BullMQ data loss (in-progress jobs marked as stalled instead of gracefully completed).

- **Spec Amendment (Attack Audit 2026-03-22):** Reference Node.js Dockerfile pattern: multi-stage build with `COPY package*.json ./` before `COPY .` for layer caching, `--mount=type=cache,target=/root/.npm` for npm cache persistence across builds (requires BuildKit), `USER node` for non-root execution, exec-form `CMD ["node", "dist/index.js"]` (shell form makes sh PID 1 which does not forward SIGTERM --- containers take 10s to stop then get SIGKILL), and `NODE_OPTIONS="--max-old-space-size=1536"` set to 75% of container memory limit. Include `.dockerignore` excluding `node_modules`, `.git`, `tests`, `*.md`, `dist` from build context to prevent 2GB+ images and layer cache invalidation on non-source changes.

- **Spec Amendment (Attack Audit 2026-03-22):** Pin Docker base images to digest, not tag: `node:20-slim@sha256:...`. Tag-based references (e.g., `node:20-slim`) are mutable --- the upstream publisher can push a different image to the same tag. Pinning to digest prevents supply chain attacks via tag mutation. Update digests quarterly or via Renovate/Dependabot Docker ecosystem updates.

- **Restart policy (INFRA-06, MED):** All services MUST have `restart: unless-stopped`. Without a restart policy, a crashed container (e.g., NATS OOM) stays down permanently until manual intervention.

- **Docker network isolation (ARCH-01, HIGH):** Production Compose files must define named networks with tier segmentation. Create three networks: `kiptra-frontend` (LB -> Hono API only), `kiptra-backend` (Hono -> Valkey, NATS, Postgres --- `internal: true`, not externally reachable), and `kiptra-monitoring` (Prometheus, Grafana, Alloy --- isolated from application traffic). Without network segmentation, a compromised frontend container can directly query Valkey or the database. Each service should only be attached to the networks it needs.

### DigitalOcean Platform Notes


- **Managed Valkey version cap (DO-03, MED):** DigitalOcean Managed Valkey replaces discontinued Managed Redis (June 2025). Wire-compatible with Redis; BullMQ and Socket.io adapters work unchanged. Plan accordingly --- verify feature parity for any Redis 7.4+ features relied upon.

- **Real client IP behind Cloudflare (DO-04, MED):** Use the `CF-Connecting-IP` header for real client IP in rate limiting and audit logs. `X-Forwarded-For` returns the Cloudflare edge IP, not the client IP, when traffic flows through Cloudflare.

- **Proxy Protocol conflict (DO-05, MED):** Proxy Protocol MUST NOT be enabled on both Cloudflare and the DigitalOcean Load Balancer simultaneously. Enable it on one or the other --- double-wrapping corrupts the connection headers.


### Rolling Deployment --- Sequential Node Update (Wave 2)


Deploy to secondary VPS first, verify health, then deploy to primary. This is a rolling deployment, not true blue-green (which requires two complete, identically-sized environments with atomic traffic switching). Constraint: All database migrations must be backward-compatible with the previous code version, since both versions run simultaneously during the rollout window. A migration compatibility check in CI blocks deploys if a migration removes or renames columns still referenced by the previous code version.


- Deploy sequence: secondary VPS first -> verify health -> primary VPS. Near-zero-downtime (brief period of mixed versions).

- Health check endpoints **(MON-03)** split into three to prevent restart loops when a dependency is briefly unreachable: (a) `/health/live` --- process liveness only, never checks external dependencies (used by Docker HEALTHCHECK and LB liveness probe), (b) `/health/ready` --- checks database connected, NATS connected, Valkey connected, Socket.io Valkey adapter connected (used by LB readiness probe --- returns 503 until all dependencies are reachable), (c) `/health/startup` --- one-time initialization check (schema migrations applied, NATS subjects created, BullMQ workers registered). A single `/health` endpoint that checks all dependencies causes restart loops when a dependency is briefly unreachable --- the LB marks the node unhealthy even though the process is fine.

- If errors spike on newly deployed node, Load Balancer automatically removes it from rotation within 90 seconds.

- WebSocket connections: existing connections drain naturally (30-second grace period). New connections route to healthy nodes.

- Deployment frequency: multiple times per day if needed. No maintenance windows.

- **Graceful shutdown (SIGTERM handler):** SIGTERM handler sequence (order is critical): (1) `worker.close()` --- stops accepting new BullMQ jobs and waits for in-progress jobs to complete (up to `lockDuration`). (2) `io.close()` --- sends disconnect to all Socket.io clients with 30-second drain. (3) Track in-flight SSE streams and send close events to trigger client reconnection to the surviving node. (4) `server.close()` --- stops accepting new HTTP connections, drains existing ones. (5) `process.exit(0)`. The BullMQ `lockDuration` must be >= 120,000ms (matching the longest LLM timeout) so in-progress jobs are not marked stalled during shutdown. This is a required Wave 2 implementation item, not an assumed behavior of the DigitalOcean Load Balancer.

> **Spec Amendment (Attack Audit 2026-03-22):** Full SIGTERM handler resource cleanup order: (1) `httpServer.close()` (stop accepting connections), (2) `worker.close()` (finish current jobs), (3) `queue.close()` (close queue connections), (4) `queueEvents.close()` (close event listeners), (5) `nc.drain()` (drain NATS --- finish in-flight, then close), (6) `io.close()` (disconnect Socket.io clients), (7) `pub.quit()` + `sub.quit()` (close Redis/Valkey clients). The current spec omits `nc.drain()` (NATS), `queue.close()`, `queueEvents.close()`, and Redis client cleanup. Without `nc.drain()`, in-flight NATS messages are dropped. Without `queue.close()`, BullMQ connection leaks on restart.

- **SSE stream drain:** On SIGTERM, track all in-flight SSE streams (agent execution streams). Send a stream close event with a retry directive (`retry: 5000`) so clients automatically reconnect to the surviving node. If an LLM inference is mid-flight, the `onStepFinish` callback should check a shutdown flag and gracefully complete the current step before yielding. Do not kill active LLM calls --- wait for the current step to finish, then close the stream.

- **Deploy-safe webhook handling:** Configure the DO Load Balancer to remove a node from rotation before deploying to it (pre-drain), not after. SendGrid retries failed webhooks for up to 72 hours, then permanently drops the email. A deploy window where the webhook endpoint returns 5xx causes lead email loss. Additionally: set up a SendGrid webhook alert --- if no inbound emails are processed for a tenant that normally receives 5+/day, page on-call. Document MX propagation delay (up to 48 hours) in the onboarding flow with a 'your email setup is pending DNS propagation' status indicator.

- **Code canary deployment:** Before promoting a deploy to both VPS nodes, deploy new code to secondary only and configure the DO Load Balancer to route 10% of traffic to secondary for a 15-minute observation window. Monitor error rate, P99 latency, and BullMQ stalled job count during the canary window. If any metric exceeds threshold, roll back secondary to the previous image. If metrics are clean, promote to primary. This is separate from the prompt canary (10% of tenants) --- code canary validates infrastructure changes, prompt canary validates AI behavior changes.


### Docker Image Registry


Docker image registry: GitHub Container Registry (ghcr.io). Image tagging: every merge to main produces an image tagged with the git SHA (`ghcr.io/kiptra-ai/backend:abc123f`). The deploy script also tags the image as 'latest'. On deploy, the outgoing image is re-tagged as 'previous'. Both 'latest' and 'previous' tags are always resident on each VPS node. Image pruning policy: keep the 5 most recent images per VPS node.

**Spec Amendment (Attack Audit 2026-03-22):** Production deploy requires: (1) `environment: production` with required reviewers configured on the GitHub Actions deploy job --- this is a GitHub-native deploy gate that prevents accidental production pushes. (2) Minimum 15-minute wait after staging deploy before production promotion (configurable via environment wait timer in GitHub repo settings). (3) Automated smoke test suite that must pass before deploy completes: health check returns 200, one agent execution completes end-to-end against a test tenant, Stripe webhook signature verification succeeds, Socket.io connection established and room join confirmed. If any smoke test fails, the deploy is automatically rolled back to the `previous` image tag.

### Testing Conventions


- **Coverage threshold:** 80% line coverage enforced in CI via Vitest coverage reporter. PRs that decrease coverage below 80% are blocked.

- **AI SDK mocking:** Use Vercel AI SDK's `MockLanguageModelV1` and `simulateReadableStream` test utilities for deterministic agent logic tests without burning real API tokens. Mock AI responses at the AIGateway adapter layer.

### Security Hardening (Wave 1 --- Non-Negotiable)


- **Headers:** HSTS, CSP, X-Frame-Options, X-Content-Type-Options --- built into Hono middleware

- **Auth:** Rate limiting on all auth endpoints (5--10 req/min/IP), account lockout after 5 failed attempts, MFA via TOTP

- **CORS:** Explicit allowlist (not wildcard) --- only Vercel frontend domain

- **Tenant isolation:** Never trust client-supplied tenant IDs --- derive from auth token server-side

- **Input validation:** Zod schemas on all API endpoints, parameterized queries everywhere

- **Spec Amendment (Attack Audit 2026-03-22):** SSRF Protection: Block private IP ranges (10.x, 172.16.x, 169.254.x, 127.x, ::1, fd00::/8) in all outbound HTTP calls from agent tool executions. Maintain an allowlist of known integration domains (JobTread, QuickBooks, Google, Microsoft Graph, SendGrid). Agents process untrusted URLs from inbound emails (SendGrid Inbound Parse) and user-submitted content --- SSRF is a real attack vector where a crafted email could cause the agent to make requests to internal services (Valkey, NATS, Postgres) or cloud metadata endpoints (169.254.169.254). Implement via a custom `fetch` wrapper used by all `IntegrationAdapter` implementations.

- **Secrets:** Environment variables only (never in code), Vercel env vars for frontend, Docker secrets for both VPS nodes

- **WAF:** Cloudflare Pro ($20/mo) in front of dual VPS (via DO Load Balancer) --- DDoS protection, bot filtering, SSL termination. **Page Rules deprecated (CF-03, MED):** Cloudflare Page Rules are deprecated as of 2025. Use Cache Rules + Configuration Rules instead for all URL-based routing and caching configuration. **WebSocket keepalive (CF-04, MED):** Socket.io `pingInterval` must be ` returns closed/filtered for all non-web ports.

- **(ARCH-01) Docker network isolation (production):** Docker Compose must define named networks with segmentation: (1) `frontend` network: Load Balancer -> Hono API containers only; (2) `backend` network: Hono -> Valkey, NATS, Postgres (internal services); (3) `monitoring` network: Prometheus, Loki, Grafana (isolated from application traffic). Without named network segmentation, all containers share the default bridge network --- a compromised frontend container can directly query Valkey and Postgres. Each service connects only to the networks it requires (e.g., Hono connects to both `frontend` and `backend`; Valkey connects only to `backend`).

- **Spec Amendment (Attack Audit 2026-03-22):** All SSE streaming endpoints MUST return `Content-Encoding: none` header to prevent Cloudflare edge compression buffering. This is SEPARATE from `X-Accel-Buffering: no` (tunnel/proxy-layer buffering fix) and applies regardless of Tunnel vs DNS proxy mode. Without this header, Cloudflare's gzip/brotli compression buffers the response body to compress it, defeating real-time token streaming --- tokens arrive as one large batch instead of incrementally.

- **Spec Amendment (Attack Audit 2026-03-22):** Cloudflare may not propagate client disconnect signals to the origin server (KB gotcha #15). Implement bidirectional heartbeat: server sends SSE keepalive comment (`: keepalive\n\n`) every 15s; monitor `res.on('close')` to detect client disconnection and call `abortController.abort()` to cancel in-flight LLM inference. Without this, LLM generation continues running (wasting tokens and compute) after the user navigates away.

- **Spec Amendment (Attack Audit 2026-03-22):** CF-06 (CONSTRAINT): Cloudflare Pro has a non-configurable 100-second proxy read timeout (KB gotcha #6). The 15s SSE keepalive interval specified above provides a 6.6x safety margin (keepalive resets the 100s timer). Do not increase keepalive interval above 90s. Enterprise plan allows 600s timeout but is not cost-justified at current scale.

- **Cloudflare SSL:** Mode = Full (Strict) --- mandatory. Install **Cloudflare Origin CA certificates** (not Let's Encrypt) on both VPS nodes and the DigitalOcean Load Balancer. **(CF-02):** Let's Encrypt renewal can fail behind Cloudflare proxy (DNS challenge required). Cloudflare Origin CA certs have 15-year validity and are purpose-built for this topology. Enable "Always Use HTTPS" and "Minimum TLS Version: 1.2". Wave 4 security audit must verify Cloudflare SSL mode is Full (Strict), not Flexible.

- **Secret rotation:** Application implements a SIGHUP handler that re-reads secrets from Docker secrets files without restarting. For immediate rotation (compromise): update Docker secret -> rolling restart (secondary first, then primary). Annual rotation schedule for all provider API keys, documented in incident response plan.

### Prompt Injection Defense (Wave 2 --- Critical)


Three-layer defense against prompt injection attacks. Agents process inbound emails, webhooks, and user input --- all are untrusted attack surfaces.


#### Layer 1 --- Architectural Defense + Input Sanitization


- **Primary defense is architectural, not classifier-based:** (a) User input is never concatenated into system prompts --- a hard structural separator (`` tag) marks untrusted content. (b) Vercel AI SDK's `system` parameter enforces separation at the framework level. (c) Agent tool permissions are statically defined --- injected prompts cannot invoke tools not in the agent's registered tool set.

- All external inputs (inbound emails via SendGrid, webhook payloads, user-submitted text) are sanitized before reaching agent prompts

- Strip control characters, limit input length, detect known injection patterns (e.g., "ignore previous instructions", "system prompt override")

- The Haiku classifier (Haiku 4.5, cost: negligible) provides a secondary probabilistic defense layer --- scores injection probability before routing to agent --- but is not the primary gate.


#### Layer 2 --- Prompt Architecture


- System prompts use delimiter-based separation (XML tags) between instructions and user content

- Agent instructions are marked as immutable; user content is explicitly sandboxed

- Agents cannot modify their own system prompts or access other agents' prompts

- Tool-use permissions are statically defined per agent --- an injected prompt cannot grant new tool access


#### Layer 3 --- Output Validation


- All agent outputs pass through a validation layer before action execution

- Structured outputs validated against JSON schemas (Zod) --- malformed responses are rejected

- Actions with side effects (sending emails, creating records, modifying data) require schema-conformant output AND human approval for client-facing actions

- Anomaly detection: if an agent's output is statistically different from its historical pattern (token count, action type, referenced entities), it is flagged for review


### AI Output Validation Pipeline (Wave 2)


Every agent output passes through a validation pipeline before execution:


`AgentResult` includes full execution trace: `{ finalOutput: T, toolCallHistory: ToolCall[], modelUsage: ModelUsageByStep[], escalationsTriggered: Escalation[] }`. The validation pipeline inspects `toolCallHistory` for anomalous patterns (unexpected tool calls, out-of-scope invocations), not just the final output.


- **Schema validation** --- output must conform to the agent's defined output schema (Zod). Malformed JSON, missing required fields, or unexpected field types -> reject and retry (max 2 retries)

- **Boundary checks** --- after 30 days of tenant operation, statistical bounds (mean &plusmn; 3 standard deviations) are computed from the tenant's own historical data per project type. For new tenants, platform-wide quartiles serve as initial reference, tightening as customer-specific data accumulates. Computed by a nightly background job per tenant. An estimate of $0 or $10M for a kitchen remodel -> flagged as anomalous

- **Action authorization** --- outputs requesting side effects (send email, create record, modify schedule) checked against the agent's permission set. Actions outside scope -> reject

- **Consistency check** --- for financial agents (Sales Agent, Bookkeeping Agent), outputs cross-referenced against source data. If the AI claims an invoice is overdue but QuickBooks shows it's paid -> flagged

- **Rate limiting** --- rate limits apply to side-effecting actions only (email sends, record creates, data modifications) --- not read-only tool calls. Tier-specific: Starter = 50 side-effecting actions/hour, Professional = 200/hour, Enterprise = 500/hour

- **Loop prevention** --- `stopWhen: stepCountIs(N)` set per agent type (Leads Agent = `stepCountIs(7)`, Sales Agent = `stepCountIs(17)`, CEO Agent = `stepCountIs(12)`) via Vercel AI SDK. **Note:** Step budgets are +2 vs. v5 values because AI SDK v6 consumes a step when generating structured output. Total token budget per execution as secondary guard. This is more permissive for legitimate multi-tool workflows and more robust against true loops than a consecutive-call heuristic.

### Database Migration Strategy


Uses the **Expand & Contract** pattern to prevent downtime:


- Add new columns/tables first (expand) --- no breaking changes

- Backfill data with migration scripts

- Update application code to use new schema

- Remove old columns later (contract)


**Critical rule:** RLS policies must NEVER reference newly-added columns until backfill is complete --- a policy referencing a NULL column silently denies all access for existing rows. Sequence: (1) add column with DEFAULT NULL, (2) backfill, (3) add NOT NULL constraint, (4) update RLS policy. Additionally, RLS policy changes affect ALL tenants simultaneously. Test every migration against a snapshot of production data before applying. Use Supabase's built-in tooling: `supabase db diff`, `supabase migration new`.



> **Spec Amendment (Attack Audit 2026-03-22):** Migration execution protocol: (1) Migrations run as a pre-deploy step on staging first, then production --- never directly to production. Staging migration must succeed and soak for at least 1 hour before production apply. (2) Use `CREATE INDEX CONCURRENTLY` always --- never `CREATE INDEX` on any table with data. Non-concurrent index creation takes a full table lock. (3) Contract phase (dropping columns/tables) requires 48-hour production soak after the code no longer references the column, not just deploy success. This catches background jobs, cron tasks, and integration callbacks that may still reference the old column. (4) Backfill strategy: small tables (< 100K rows) = migration script with batch updates (1000 rows/batch, 100ms sleep between batches). Large tables (>= 100K rows) = BullMQ background job with progress tracking, resumable on failure. (5) Every migration must have a corresponding rollback script in `supabase/migrations/rollback/` --- see Database Rollback amendment below.

**Column addition convention:** All new columns on existing tables must be nullable with `DEFAULT NULL`. Never add a non-null column with a default value to a high-traffic table (e.g., `agent_executions`) --- this rewrites the entire table and takes a full table lock. Non-null constraints and default values must be added in a separate migration after the column is backfilled. This is enforced by CI lint rule.



> **Spec Amendment (Attack Audit 2026-03-22):** Database rollback protocol: Every migration MUST have a corresponding rollback script in `supabase/migrations/rollback/` named with the same timestamp prefix. Rollback categories: (1) Additive (added column/table) = `DROP COLUMN`/`DROP TABLE`. (2) Data migration (backfill) = revert backfill script that restores previous values from a pre-migration snapshot table. (3) Destructive (dropped column that had data) = restore from Supabase backup (RPO: 24h pre-PITR, minutes post-PITR). CI lint rule: reject any migration PR that lacks a corresponding rollback file. Add data integrity checks to canary observation window: row counts on affected tables (delta > 5% = alert), null rate on backfilled columns (any increase = alert), RLS policy verification (run adversarial cross-tenant query against canary), foreign key constraint validation (`pg_constraint` check for orphaned rows).

### Error Taxonomy (Platform-Wide)


Standardized error codes across all API endpoints, agent executions, and integration callbacks. Every error includes a machine-readable `code`, human-safe `message`, HTTP status, retry guidance, and correlation IDs for support debugging.


#### Base Error Interface


TypeScript
interface ApiError {
 code: string; // e.g., &quot;AGENT_EXECUTION_TIMEOUT&quot;
 message: string; // Human-readable, safe to display to end users
 statusCode: number; // HTTP status code
 details?: unknown; // Additional context (never contains PII)
 retryable: boolean; // Whether the client should retry
 retryAfterMs?: number; // Suggested retry delay if retryable
 documentationUrl?: string;
 requestId: string; // Correlation ID for support tickets
 traceId?: string; // OpenTelemetry trace ID for debugging
}
```


**Error code format:** `DOMAIN_TYPE_DETAIL` --- e.g., `AUTH_SESSION_EXPIRED`, `AGENT_EXECUTION_TIMEOUT`.
